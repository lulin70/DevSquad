## 二、方案设计

### 2.1 核心思路：代码走读增强包

创建第4个关注点增强包 `templates/concerns/code-walkthrough.yaml`，定义7个角色的代码走读视角。

**为什么用增强包而不是独立模块？**
- 复用现有的 ConcernPackLoader 自动匹配机制
- 与权限/API/数据管道增强包可组合
- 无需修改核心调度逻辑
- 遵循"关注点增强"而非"项目模板"的设计理念

### 2.2 七角色代码走读视角定义

| 角色 | 走读视角 | 核心关注点 | 输出物 |
|------|---------|-----------|--------|
| 🏗️ **架构师** | 架构合理性 | 模块划分、依赖方向、分层是否清晰、循环依赖、接口抽象度 | 架构走读报告 |
| 🔒 **安全专家** | 安全漏洞 | 输入验证、认证授权、数据保护、注入风险、敏感信息泄露、合规性 | 安全走读报告 |
| 🧪 **测试专家** | 测试覆盖 | 测试完整性、边界条件、异常路径、测试独立性、Mock合理性 | 测试走读报告 |
| 💻 **独立开发者** | 代码质量 | 命名规范、函数长度、圈复杂度、重复代码、错误处理 | 代码质量走读报告 |
| ⚙️ **运维专家** | 运维可行性 | 部署复杂度、配置管理、日志规范、监控埋点、容灾能力 | 运维走读报告 |
| 📋 **产品经理** | 需求覆盖 | 需求实现完整性、用户故事覆盖、验收标准达成、边界场景 | 需求走读报告 |
| 🎨 **UI设计师** | 交互一致性 | 组件复用、交互规范、响应式、无障碍、视觉一致性 | 交互走读报告 |

### 2.3 性能审查归属

**方案**: 性能审查由**架构师**主责 + **运维专家**辅责

| 性能维度 | 主责角色 | 辅责角色 | 说明 |
|---------|---------|---------|------|
| 算法复杂度 | 架构师 | 开发者 | 时间/空间复杂度评估 |
| 数据库性能 | 架构师 | 运维专家 | 慢查询、索引、连接池 |
| 并发性能 | 架构师 | 运维专家 | 线程安全、锁竞争、连接管理 |
| 资源使用 | 运维专家 | 架构师 | CPU/内存/磁盘/网络 |
| 可扩展性 | 架构师 | 运维专家 | 水平扩展、负载均衡 |
| 响应时间 | 运维专家 | 架构师 | P50/P95/P99、SLA |

**理由**:
- 架构师负责"设计层面"的性能（算法、数据结构、架构模式）
- 运维专家负责"运行层面"的性能（资源、监控、容量规划）
- 两者互补，不遗漏

### 2.4 共识投票机制

代码走读结果通过**五轴共识引擎**投票决定：

| 轴 | 权重 | 主责角色 | 说明 |
|----|------|---------|------|
| Correctness | 0.25 | 开发者 + 测试专家 | 逻辑正确、无Bug |
| Security | 0.25 | 安全专家 | 漏洞、合规 |
| Architecture | 0.20 | 架构师 | 设计模式、模块化 |
| Operability | 0.15 | 运维专家 | 部署、监控、容灾 |
| Readability | 0.15 | 开发者 + 全角色 | 代码清晰、可维护 |

**与现有五轴的区别**: 将 Performance 轴替换为 Operability 轴，因为：
- Performance 是 Architecture + Operability 的子集
- Operability 更全面（包含部署、监控、容灾，不仅仅是性能）
- 运维专家在原五轴中没有明确位置

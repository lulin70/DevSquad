## 一、评估背景与方法论

### 1.1 评估动机

Matt Pocock (TypeScript 社区知名专家) 开源了其日常使用的 21 个 .claude skills，覆盖软件研发全流程。本评估旨在判断哪些 skill 的方法论、设计理念或实现可直接为 DevSquad V4.0.11 (185+ modules, 6 atomic sub-skills, 11-phase lifecycle) 借鉴、学习或调用。

### 1.2 四级分类标准

| 分类 | 定义 | 落地方式 |
|------|------|----------|
| **借鉴 (Borrow)** | 方法论/工作流思路可融入 DevSquad 现有模块 | 修改现有模块代码 + 同步文档 |
| **学习 (Learn)** | 设计理念/原则影响 DevSquad 未来演进方向 | 记录到 ROADMAP / Less Learned |
| **调用 (Call)** | 可作为外部 skill 通过 PluginHotLoader 直接挂载 | 注册到 BUILTIN_PLUGINS |
| **不适用 (N/A)** | 与 DevSquad 现有能力重复或不匹配 DevSquad 定位 | 不采纳 |

### 1.3 评估流程

```
Matt 21 skills → SKILL.md 拉取 (15/21 detailed) → DevSquad 现有模块映射
```

## 四、详细评估（按技能）

### 4.1 强借鉴 (6 个) — 优先级 P0

#### 4.1.1 `tdd` — 测试驱动开发 (red → green, NO refactor)

**Matt 核心理念**:
- 红 → 绿循环，**移除 refactor 阶段** (refactor 属于 review 阶段，不属于 TDD loop)
- **Vertical slice** — 一次一个 seam、一个测试、一个最小实现
- **Seams-up-front** — 写测试前先与用户确认 seam
- **反模式**: Implementation-coupled / Tautological / Horizontal slicing

**DevSquad 现状对比**:
- DevSquad 有 `TestQualityGuard` (API validation / anti-pattern detection / dimension coverage) + Iron Rule 2 (失败即报告，禁止改断言通过)
- DevSquad Iron Rule 3 (维度完整性) 已覆盖 happy/error/boundary/performance/config/integration/security

**借鉴点**:
1. **"移除 refactor 阶段"** — 与 DevSquad Iron Rule 2 呼应，refactor 应在 `review` skill (P8 代码审查) 而非 TDD loop 内
2. **Vertical slice tracer bullet** — 每次 cycle 一个端到端切片，可强化 `MicroTaskPlanner` (2-5 min decomposition)
3. **Seams-up-front 确认** — 可加入 `TestQualityGuard` 的 pre-audit 阶段
4. **Tautological 反模式** — `expect(add(a,b)).toBe(a+b)` 永远通过，DevSquad 可在 anti-pattern detection 中新增此检测

**7 角色共识**: **强借鉴** (加权 8.15)
- Architect: vertical slice 强化 MicroTaskPlanner 的切片粒度
- Tester: tautological 反模式检测是 TestQualityGuard 的有力补充
- Coder: seams-up-front 减少测试返工

**落地建议**: 在 `test_quality_guard.py` 新增 `detect_tautological_test()` + `verify_seams_confirmed()` 方法

---

#### 4.1.2 `domain-modeling` — 领域模型在线维护 (CONTEXT.md + ADR)

**Matt 核心理念**:
- `CONTEXT.md` 是**纯术语表**，禁止包含实现细节
- ADR 三准则: Hard-to-reverse + Surprising-without-context + Real-tradeoff (三条全满足才写 ADR)
- **Inline update** — 术语确定时立即更新 CONTEXT.md，不批量
- 挑战术语冲突 / 锐化模糊语言 / 讨论具体场景 / 交叉引用代码

**DevSquad 现状对比**:
- DevSquad 有 `MCEAdapter` (CarryMem 集成) + `project_memory` + `docs/spec/SPEC.md` + 11-phase P1(Requirements)
- DevSquad 的"活文档原则"与此高度契合，但缺乏**纯术语表**机制

**借鉴点**:
1. **CONTEXT.md as pure glossary** — DevSquad 的 SPEC.md 混合了术语+实现+架构，可拆分出 `docs/spec/GLOSSARY.md` 纯术语表
2. **ADR 三准则** — DevSquad 的 docs/audits/ 目录有评估文档但缺乏 ADR 体系，可建立 `docs/adr/` 目录
3. **Inline update discipline** — 强化 DevSquad 的"活文档"执行力度
4. **CONTEXT-MAP.md 多上下文** — DevSquad 185+ 模块可按子系统 (collaboration/api/qa/loop_engineering/autonomous/plugins) 拆分 CONTEXT

**7 角色共识**: **强借鉴** (加权 8.05)
- Architect: ADR 体系补全架构决策追溯链
- PM: GLOSSARY.md 锐化需求文档精确度
- Coder: 术语统一减少代码命名歧义

**落地建议**: 创建 `docs/spec/GLOSSARY.md` + `docs/adr/` 目录 + 在 `role_skill_loader.py` 中加载 GLOSSARY

#### 4.1.3 `improve-codebase-architecture` — 架构深化扫描 + HTML 报告

**Matt 核心理念**:
- **Deepening opportunities** — 将 shallow module 转为 deep module
- **Deletion test** — 想象删除模块，复杂度消失=pass-through，复杂度重现=earning its keep
- **HTML 可视化报告** — Tailwind + Mermaid，before/after 对比图
- **Grilling loop** — 用户选定候选后进入访谈式深化
- 词汇: Module / Interface / Implementation / Depth / Seam / Adapter / Leverage / Locality

**DevSquad 现状对比**:
- DevSquad 有 `TechDebtManager` + `CodebaseDebtScanner` + `YagniChecker` + `RedesignAuditor` (YAGNI/STDLIB/DUPLICATE/OVERENGINEERING)
- DevSquad 有 `CodeKnowledgeGraph` (SQLite-backed symbols/edges/files)

**借鉴点**:
1. **Deletion test** — `RedesignAuditor` 可新增 deletion-test 检测维度
2. **Deep/Shallow module 词汇** — 强化 `CodeKnowledgeGraph` 的模块分析维度
3. **HTML 可视化报告** — DevSquad 的 `StreamlitDashboard` 可新增"架构深化机会"视图
4. **Grilling loop 联动** — 架构审查后接入 grilling 访谈 (借鉴 grilling skill)

**7 角色共识**: **强借鉴** (加权 8.05)
- Architect: deletion test 是 SRP 分析的有力补充 (避免 God Class 误判)
- Coder: deep/shallow 词汇统一架构讨论语言
- UI: HTML 报告的 before/after 可视化模式可借鉴

**落地建议**: `redesign_auditor.py` 新增 `deletion_test()` 方法 + Dashboard 新增"Architecture Deepening"视图

#### 4.1.4 `diagnosing-bugs` — 6 阶段调试循环

**Matt 核心理念**:
- **Phase 1 (Build feedback loop) 是核心** — 没有 red-capable command 就不进入 Phase 2
- **Red-capable criterion**: on-Red-capable + on-Deterministic + on-Fast + on-Agent-runnable
- **3-5 ranked hypotheses** — 单假设会锚定，必须生成多个可证伪假设
- **[DEBUG-xxx] tag** — 所有调试日志打唯一前缀，cleanup 时一次 grep 清除
- **Phase 6 post-mortem** — 记录正确假设到 commit message

**DevSquad 现状对比**:
- DevSquad 有 `Tester` role + `TestQualityGuard` + `ExecutionGuard` + `FeedbackControlLoop` (Sense→Decide→Act→Feedback)
- DevSquad 有 `RetrospectiveEngine` (post-task analysis)

**借鉴点**:
1. **Red-capable command 强制门禁** — DevSquad 的 `VerificationGate` 可新增 "debug-loop-ready" 检查
2. **3-5 ranked hypotheses** — `FeedbackControlLoop` 的 Decide 阶段可强制多假设生成
3. **[DEBUG-xxx] tag cleanup** — `ExecutionGuard` 可新增调试日志标记与清理机制
4. **Phase 1 优先级最高** — DevSquad Tester role 的 SKILL.md 可吸收"先建反馈循环再调试"原则

**7 角色共识**: **强借鉴** (加权 7.85)
- Tester: red-capable criterion 是调试纪律的核心
- Coder: [DEBUG-xxx] tag 解决调试日志残留问题
- Architect: Phase 1 门禁防止"盲目假设"浪费

**落地建议**: `verification_gate.py` 新增 `verify_debug_loop_ready()` + Tester role SKILL.md 吸收 6-phase 调试法

#### 4.1.5 `codebase-design` — Deep Module 设计词汇

**Matt 核心理念**:
- **Deep module** = small interface + lots of implementation (高 leverage + 高 locality)
- **Seam** (Michael Feathers) — 不编辑原地即可改变行为的位置
- **One adapter = hypothetical seam. Two adapters = real seam.**
- **The interface is the test surface.**
- **Deletion test** — 删除模块后复杂度消失=pass-through
- 拒绝 Ousterhout 的 "depth as ratio of impl-lines to interface-lines" (会奖励填充实现)

**DevSquad 现状对比**:
- DevSquad 有 `YagniChecker` (6 levels) + `RedesignAuditor` + `CodeKnowledgeGraph`
- DevSquad 有 God Class 教训: SRP 分析优于机械阈值 (52 候选 → 1.9% hit rate)

**借鉴点**:
1. **Deep/Shallow 词汇** — 统一 DevSquad 架构讨论语言 (替代 God Class/复杂度等模糊术语)
2. **One-adapter-two-adapters 准则** — `YagniChecker` 可新增 "premature seam" 检测
3. **Interface-is-test-surface** — 强化 `TestQualityGuard` 的 seam 验证
4. **拒绝 lines-ratio** — 印证 DevSquad God Class 教训，记录到 Lessons Learned

**7 角色共识**: **强借鉴** (加权 7.95)
- Architect: deep/shallow 词汇是 SRP 分析的精确化
- Tester: interface-is-test-surface 强化 seam 验证
- Coder: one-adapter-two-adapters 防止过度抽象

**落地建议**: `yagni_checker.py` 新增 `check_premature_seam()` + 架构师 role SKILL.md 吸收 deep-module 词汇

#### 4.1.6 `writing-great-skills` — Skill 写作元参考

**Matt 核心理念**:
- **Predictability is the root virtue** — skill 的目标是"每次运行走相同流程"，而非"产生相同输出"
- **Model-invoked vs User-invoked trade-off**: model-invoked 付 context load 省 cognitive load；user-invoked 反之
- **Information hierarchy**: in-skill step → in-skill reference → external reference (progressive disclosure)
- **Leading words** — 复用模型预训练中的紧凑概念 (如 "tight loop" / "red-capable")
- **Failure modes**: premature completion / duplication / sediment / sprawl / no-op
- **No-op test** — 这行话改变默认行为吗？不改就是 no-op

**DevSquad 现状对比**:
- DevSquad 有 `StandardizedRoleTemplate` (V2 SKILL.md anatomy: overview/when_to_use/process_steps/rationalizations/red_flags/verification_requirements)
- DevSquad 有 `Skillifier` (auto-generate skills) + `RoleSkillLoader` + 6 atomic sub-skills

**借鉴点**:
1. **Model-invoked vs User-invoked 分类** — DevSquad 的 6 atomic sub-skills 目前都是 model-invoked，可考虑为 dispatch/intent 增加 user-invoked 快捷方式
2. **Leading words** — DevSquad SKILL.md 可引入 "consensus" / "gate" / "anchor" 等 leading words 锐化
3. **No-op test** — `Skillifier` 自动生成 skill 时可应用 no-op 检测
4. **Failure modes 词汇** — 强化 `StandardizedRoleTemplate` 的质量审计维度
5. **Progressive disclosure** — DevSquad SKILL.md 当前较扁平，可按 information hierarchy 重构

**7 角色共识**: **强借鉴** (加权 7.65)
- Architect: progressive disclosure 优化 skill 结构
- Coder: no-op test 提升 Skillifier 生成质量
- PM: failure modes 词汇统一 skill 质量讨论

**落地建议**: `standardized_role_template.py` 新增 `apply_no_op_test()` + `classify_invocation_type()` 方法

#### 4.1.7 `grilling` — 可复用访谈循环

**Matt 核心理念**:
- **One question at a time** — 多问题并发让用户困惑
- **Recommended answer per question** — 每个问题附推荐答案
- **Explore codebase instead of asking** — 能从代码找答案就不问用户
- **Walk down each branch of design tree** — 决策树逐分支解决

**DevSquad 现状对比**:
- DevSquad 有 `RuleCollector` (intent detection / rule extraction / sanitization / CarryMem+JSON storage)
- DevSquad 有 `InputValidator` (53 patterns) + `PromptAssembler`

**借鉴点**:
1. **One-question-at-a-time** — `RuleCollector` 的访谈模式可吸收此纪律
2. **Recommended-answer** — `PromptAssembler` 可为每轮访谈注入推荐答案
3. **Explore-before-ask** — `CodeKnowledgeGraph` 可在访谈前自动查询代码
4. **Design-tree-walk** — `WorkflowEngine` 的 P1(Requirements) 阶段可引入决策树遍历

**7 角色共识**: **强借鉴** (加权 7.95)
- PM: one-question-at-a-time 提升需求访谈质量
- Coder: explore-before-ask 减少用户负担
- Architect: design-tree-walk 结构化决策

**落地建议**: `rule_collector.py` 新增 `grilling_mode()` (one-question-at-a-time + recommended-answer) + `code_explore_before_ask()`

1. **借鉴方法论，不照搬实现** — Matt skills 是 Markdown 指令，DevSquad 是 Python 代码，借鉴的是设计理念

3. **文档同步** — 每个借鉴项落地时同步更新 SPEC.md / SKILL.md / CHANGELOG / GLOSSARY.md (新增)

## 一、当前架构 (V3.7.2)

### 1.1 架构概览

DevSquad V3.7.2 采用**分层 + 事件驱动**架构，核心是 18-20 步的 dispatch 管线，由 `MultiAgentDispatcher` 统一编排。

### 1.4 当前架构的不足（V3.8 要解决的）

| 不足 | 影响 | V3.8 解决方案 |
|------|------|-------------|
| 代码审查单阶段，规范符合性与质量审查混合 | 审查反馈模糊，P0问题可能漏过 | #2 两阶段审查 |
| security/tester发现问题后无自动修复闭环 | 问题依赖人工处理，MTTR长 | #3 严重度路由+自动修复 |
| LLM缓存键非SHA-256，退避无jitter | 键冲突+高并发重试风暴 | #9 缓存退避强化 |
| ConsensusEngine仅加权投票，无去重/冲突解决 | 重复讨论+意见拉锯 | #4 Judge Agent |
| WorkflowStep未区分确定性/LLM节点 | 可审计性弱，测试策略模糊 | #6 步骤分离 |
| 任务粒度粗，无微任务约束 | AI输出"方向"而非"可执行步骤" | #7 微任务粒度 |

## 二、目标架构 (V3.8)

### 2.1 架构演进原则

1. **增量演进**: 不重写，在现有模块上增强
2. **组合优先**: 新功能以独立模块 + 组合注入方式接入，避免 God Class 回归
3. **事件驱动**: 新增模块通过 `EventBus` 解耦，不直接侵入 dispatch 管线
4. **渐进启用**: 新功能默认关闭，配置开关控制
5. **可观测**: 所有新模块接入 `PrometheusMetrics`

### 4.3 关键数据流变化

#### 变化1: 两阶段审查插入点

```
V3.7.2:  Step 9 (post_exec) ──→ Step 10 (consensus)
V3.8:    Step 9 (post_exec) ──→ [Step 9.5 两阶段审查] ──→ Step 10 (consensus)
                                    │
                                    ├─ 阶段1失败 → 阻断 (不进入Step 10)
                                    └─ 阶段2 CRITICAL/HIGH → 阻断
```

#### 变化2: 严重度路由 + 自动修复闭环

```
V3.7.2:  Step 10 (consensus) ──→ Step 11 (permission)
                              (问题依赖人工处理)

V3.8:    Step 10 (consensus) ──→ [Step 10.5 SeverityRouter + AutoFixLoop] ──→ Step 11
                                    │
                                    ├─ HIGH问题 → AutoFixLoop
                                    │   ├─ 修复 → 回归 → 通过 → 继续
                                    │   └─ 3轮失败 → 升级人工
                                    └─ MEDIUM/LOW → 记录, 不阻断
```

#### 变化3: Judge Agent 共识预处理

```
V3.7.2:  Worker意见 ──→ ConsensusEngine.reach_consensus() → 加权投票结果

V3.8:    Worker意见 ──→ JudgeAgent预处理 ──→ ConsensusEngine.reach_consensus()
                          │
                          ├─ 去重: 合并相似意见
                          ├─ 冲突解决: 裁决冲突
                          ├─ 置信度: <0.5标记
                          └─ 历史(可选): 先例参考
```

#### 变化4: 缓存键统一 + 退避抖动

```
V3.7.2:  LLM调用 ──→ LLMCache.get(legacy_key) ──→ miss ──→ LLM API
                                                          │
                                                          ├─ 失败 ──→ LLMRetry (指数退避, 无jitter)
                                                          └─ 成功 ──→ LLMCache.set(legacy_key)

V3.8:    LLM调用 ──→ CacheKeyStrategy.make_key(SHA-256) ──→ 敏感度检测
                                                          │
                                                          ├─ 高敏感 → 不缓存/加密缓存
                                                          └─ 正常 ──→ LLMCache.get(sha256_key)
                                                                      │
                                                                      ├─ hit → 返回
                                                                      └─ miss ──→ LLM API
                                                                                  │
                                                                                  ├─ 失败 ──→ LLMRetry (指数退避 + jitter)
                                                                                  └─ 成功 ──→ LLMCache.set(sha256_key)
```

#### 变化5: 微任务拆分

```
V3.7.2:  Coordinator.plan_task() ──→ ExecutionPlan (粗粒度步骤)

V3.8:    Coordinator.plan_task() ──→ MicrotaskPlanner (P8阶段)
                                      │
                                      ├─ 拆分为2-5分钟微任务
                                      ├─ 每个含: file_paths + verification_cmd
                                      └─ 上限20个
                                      │
                                      ▼
                                   ExecutionPlan (含微任务 + 验证命令)
```

## 五、向后兼容性

### 5.1 兼容性原则

V3.8 严格保证向后兼容：
1. **API 不变**: `MultiAgentDispatcher.dispatch()` 签名不变
2. **CLI 不变**: 现有 CLI 命令行为不变
3. **REST 不变**: 现有端点响应结构不变（新增字段为可选）
4. **数据兼容**: 现有 checkpoint/缓存自动迁移
5. **配置兼容**: 新配置项有默认值，旧配置文件可用

### 5.2 各增强的兼容性方案

#### #2 两阶段审查
- **默认行为**: `two_stage_review.enabled=false`，退化为现有 `VerificationGate` 单阶段
- **启用方式**: 用户配置 `enabled=true` 后生效
- **API 兼容**: `DispatchResult` 新增 `review_stages` 字段为可选，旧消费者忽略即可

#### #3 严重度路由 + 自动修复
- **默认行为**: `severity_router.enabled=false`，问题处理退化为人工（现状）
- **启用方式**: `enabled=true` + `auto_fix_mode="notify"`（仅通知）或 `"auto_fix"`（自动修复）
- **API 兼容**: `DispatchResult` 新增 `auto_fix_log` 字段为可选

#### #9 缓存退避
- **键迁移**: 旧缓存键自动失效（不主动迁移，自然过期），新调用使用 SHA-256 键
- **配置兼容**: `cache.key_strategy` 默认 `"sha256"`，可配置 `"legacy"` 回退
- **Redis 兼容**: `enable_redis_cache` 生产环境默认 True，开发环境默认 False（保持现状）

#### #4 Judge Agent
- **默认行为**: `judge_agent.enabled=false`，退化为现有 `ConsensusEngine` 纯加权投票
- **历史学习**: `history_learning=false` 默认关闭
- **API 兼容**: `ConsensusRecord` 新增字段为可选

#### #6 步骤分离
- **字段兼容**: `WorkflowStep.node_type` 默认 `"hybrid"`，旧数据反序列化自动填充默认值
- **序列化兼容**: `to_dict()` / `from_dict()` 处理缺失字段
- **指标兼容**: Prometheus 新增 `node_type` label，旧查询不受影响（label 缺失时为空）

#### #7 微任务粒度
- **默认行为**: `microtask.enabled=true`（P8阶段默认启用），但其他阶段不受影响
- **禁用方式**: `microtask.enabled=false` 退化为粗粒度任务
- **API 兼容**: `WorkflowStep` 新增 `file_paths` / `verification_cmd` 为可选字段

### 5.3 数据迁移

| 数据类型 | 迁移策略 | 影响 |
|---------|---------|------|
| LLM 缓存（旧键） | 自然过期（TTL），不主动迁移 | 无（缓存miss后重新生成） |
| Checkpoint（旧格式） | `from_dict()` 兼容缺失字段，填充默认值 | 无（自动兼容） |
| WorkflowStep（旧格式） | `node_type` 默认 `"hybrid"`，`file_paths` 默认 `[]` | 无（自动兼容） |
| ConsensusRecord（旧格式） | 新增字段默认 None/空 | 无（自动兼容） |
| .devsquad.yaml（旧配置） | 新配置项有默认值，旧配置文件可用 | 无（缺失项用默认值） |

### 5.4 回滚方案

每个增强可独立回滚（详见 PRD 5.4 节）。整体回滚步骤：

```
1. 配置所有 v3_8.*.enabled = false
2. 验证 dispatch 行为退化为 V3.7.2
3. （可选）回退 pyproject.toml 版本号到 3.7.2
4. 现有数据无需清理（新字段为可选，旧消费者忽略）
```

### 5.5 废弃计划（Deprecation）

V3.8 不废弃任何现有 API。以下为未来版本（V3.9/V4.0）的废弃预告：

| 模块/方法 | 废弃版本 | 替代方案 | 备注 |
|----------|---------|---------|------|
| `LLMCache` 旧键格式 | V3.9 | SHA-256 键 | V3.8 保持兼容，V3.9 发出 DeprecationWarning |
| `ConsensusEngine` 无 JudgeAgent 模式 | V4.0 | JudgeAgent 增强模式 | V3.8/V3.9 保持兼容 |
| RBAC/Audit Preview 状态 | V3.9 | 正式集成 | 配合 #10 本地优先产物 |

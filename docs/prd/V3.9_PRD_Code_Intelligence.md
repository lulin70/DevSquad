### V39-01: 代码知识图谱 (CodeKnowledgeGraph)

**优先级**: P0
**来源**: colbymchenry/codegraph
**目标**: 将 CodeMapGenerator 从一次性 AST 分析升级为持久化代码知识图谱

**背景**: 当前 CodeMapGenerator 每次调用都重新扫描代码，Worker 执行任务时需要 Read/Grep 文件来理解代码结构。CodeGraph 证明预索引可减少 58% 工具调用。

**功能需求**:

1. **持久化存储**: 将代码结构（函数、类、模块、依赖关系、调用图）存储为 SQLite 持久化图谱
2. **增量更新**: 文件变更时只更新受影响的节点，不全量重建
3. **查询接口**: 提供以下查询能力：
   - `find_callers(function_name)` → 查找谁调用了某函数
   - `find_callees(function_name)` → 查找某函数调用了什么
   - `find_dependencies(module_path)` → 查找模块依赖
   - `find_symbol(symbol_name)` → 查找符号定义位置
   - `get_call_graph(entry_point)` → 获取调用图
   - `find_similar_implementations(signature)` → 查找相似实现（折叠冗余）
4. **自动同步**: 可选的 watchdog 文件监听，代码变更时自动更新图谱
5. **与现有模块融合**:
   - CodeMapGenerator 成为 CodeKnowledgeGraph 的构建器
   - Worker 执行前先查询图谱获取上下文，减少 Read/Grep
   - DualLayerContextManager 的项目级上下文从图谱获取
   - PerformanceFingerprint 可用图谱做相似任务匹配

**验收标准**:
- 图谱构建后，Worker 工具调用减少 ≥30%
- 图谱查询响应时间 <50ms
- 增量更新 <500ms（单文件变更）
- 现有 CodeMapGenerator 测试全部通过（向后兼容）

**与现有模块的融合方案**:

```
CodeMapGenerator (现有, 保留)
    ↓ 构建图谱
CodeKnowledgeGraph (新)
    ├── 存储: SQLite (codegraph.db)
    ├── 索引: 符号表 + 调用图 + 依赖图
    ├── 查询: CodeGraphQuery API
    └── 同步: 可选 watchdog 文件监听
         ↓
    ┌────┴────────────────┐
    ↓                     ↓
Worker (查询替代Read)   MCP Server (暴露工具)
    ↓                     ↓
DualLayerContext       codegraph_explore
(项目级上下文)          (外部agent查询)
```

### V39-03: YAGNI 梯子检查

**优先级**: P1
**来源**: DietrichGebert/ponytail
**目标**: MicroTaskPlanner 分解任务前先检查每个微任务的必要性

**背景**: Ponytail 的 YAGNI 梯子证明 80-94% 的代码可以减少。DevSquad 的 MicroTaskPlanner 当前只分解任务，不检查"这个微任务需要做吗？"

**功能需求**:

1. 新增 `YagniChecker` 类:
   - `check(task: MicroTask) -> YagniResult`: 检查微任务是否必要
   - 检查规则（Ponytail 梯子）:
     1. **需要存在吗？** → 如果是纯探索性任务且无明确产出，标记 SKIP
     2. **标准库能做？** → 如果 Python 标准库已有，标记 USE_STDLIB
     3. **已有依赖能做？** → 如果项目已安装的依赖能做，标记 USE_DEPENDENCY
     4. **一行能写完？** → 如果是简单配置/调用，标记 ONE_LINER
     5. **最后才写最少代码** → 标记 MINIMAL
   - 每个标记附带升级路径（`upgrade_path` 字段）
2. MicroTaskPlanner 集成:
   - `plan()` 方法在生成微任务后，对每个微任务运行 YagniChecker
   - SKIP 的微任务不执行
   - USE_STDLIB/USE_DEPENDENCY 的微任务附带提示
3. **"偷懒但不疏忽"原则**:
   - 以下情况永远不标记 SKIP:
     - 安全验证（输入验证、权限检查）
     - 错误处理（异常捕获、边界条件）
     - 数据丢失防护
     - 无障碍检查

**验收标准**:
- YagniChecker 对 10 个典型微任务的判断准确率 ≥80%
- 被标记 SKIP 的微任务确实不必要（人工验证）
- 安全/错误处理/数据丢失任务永远不会被标记 SKIP
- 每个偷懒决策有 `shortcut:` 标记和升级路径

### V39-04: 三维度 Prompt 调节

**优先级**: P1
**来源**: Leonxlnx/taste-skill
**目标**: 用三个直观维度替代当前的 3 变体 + 5 风格

**背景**: taste-skill 用 VARIANCE/MOTION/DENSITY 三个维度控制 UI 生成，比 DevSquad 当前的 3 变体（concise/balanced/detailed）+ 5 风格更直观。

**功能需求**:

1. 新增 `PromptDials` 类:
   - `VERBOSITY` (1-5): 1=极简, 3=平衡, 5=详尽 — 控制输出长度和细节
   - `CREATIVITY` (1-5): 1=保守, 3=平衡, 5=创新 — 控制方案探索范围
   - `RISK_TOLERANCE` (1-5): 1=最安全, 3=平衡, 5=激进 — 控制风险偏好
2. PromptAssembler 集成:
   - `assemble()` 方法接受可选的 `dials: PromptDials` 参数
   - 根据 dials 值动态调整 prompt:
     - VERBOSITY=1 → "用最少代码回答，不要解释"
     - VERBOSITY=5 → "提供完整实现+注释+测试用例+替代方案"
     - CREATIVITY=1 → "使用最成熟的方案，不要探索"
     - CREATIVITY=5 → "探索3+种方案，包括非传统方案"
     - RISK_TOLERANCE=1 → "优先安全，使用经过验证的方案"
     - RISK_TOLERANCE=5 → "可以使用实验性技术，接受更高风险"
3. **向后兼容**: 不传 dials 时，默认 VERBOSITY=3/CREATIVITY=3/RISK_TOLERANCE=3（等价于当前 balanced）
4. Dispatcher 集成:
   - `dispatch()` 方法接受可选的 `dials` 参数
   - 传递给 PromptAssembler → Worker
5. CLI 集成:
   - `--verbosity N --creativity N --risk N` 参数

**验收标准**:
- 三维度调节可独立工作，互不干扰
- 默认值等价于当前 balanced 变体（向后兼容）
- 不同 dials 值产生明显不同的 prompt
- CLI 参数可正确传递到 dispatch

### V39-05: Redesign Audit 第三阶段

**优先级**: P1
**来源**: Leonxlnx/taste-skill
**目标**: TwoStageReviewGate 增加第三阶段"精简审查"

**背景**: 当前 TwoStageReviewGate 有两个阶段：spec compliance + code quality。taste-skill 的 redesign-audit 协议启发我们增加第三阶段：审查代码是否可以更简洁。

**功能需求**:

1. TwoStageReviewGate 新增第三阶段 `ReviewStage.REDESIGN`:
   - 检查项:
     - **YAGNI 检查**: 是否有不必要的代码/函数/类？
     - **标准库检查**: 是否有可以用标准库替代的自定义实现？
     - **重复检查**: 是否有重复代码可以抽取？
     - **过度工程检查**: 是否有过度抽象/不必要的配置项？
     - **冗余依赖检查**: 是否引入了不必要的依赖？
   - 产出: `RedesignFinding` 列表，每项包含 `current` (当前代码) + `suggested` (建议精简方案) + `saving_lines` (节省行数)
2. 严重度分级:
   - CRITICAL: 有 >50 行代码可以删除
   - HIGH: 有不必要的依赖或过度工程
   - MEDIUM: 有重复代码可以抽取
   - LOW: 有小的简化空间
3. **与 SeverityRouter 融合**: Redesign Audit 的 finding 也进入 SeverityRouter 处理
4. **向后兼容**: 第三阶段默认启用，可通过 `enable_redesign_audit=False` 关闭

**验收标准**:
- 第三阶段能发现至少 80% 的冗余代码（人工验证）
- RedesignFinding 包含具体的精简建议
- 不影响现有两阶段审查的功能
- 可通过参数关闭

### V39-06: RBAC/Audit 集成到 dispatch pipeline

**优先级**: P1
**来源**: V3.8 项目评估 P0 项
**目标**: 将 V3.6.5 的 Preview 状态 RBAC/Audit 模块集成到 dispatch pipeline

**功能需求**:

1. **RBAC 集成**:
   - Dispatcher 初始化时接受可选的 `rbac_engine` 参数
   - `dispatch()` 方法在执行前检查用户权限:
     - 用户是否有权使用请求的 roles？
     - 用户是否有权执行请求的 mode？
     - 用户是否有权访问目标项目？
   - 权限检查失败时返回 `DispatchResult(success=False, error="RBAC: ...")`
2. **Audit Logger 集成**:
   - Dispatcher 在以下事件记录审计日志:
     - dispatch 开始（who/what/when）
     - dispatch 结束（duration/success/roles）
     - 权限拒绝（who/what/why）
     - 错误发生（error type/context）
   - 审计日志存储在 SQLite（复用 HistoryManager）
3. **向后兼容**: 不配置 RBAC/Audit 时，行为与当前完全一致

**验收标准**:
- RBAC 权限检查正确拦截未授权请求
- Audit 日志完整记录 dispatch 生命周期
- 不配置时无性能影响
- 现有测试全部通过

### V39-07: 多宿主适配层

**优先级**: P2
**来源**: Ponytail (8 平台) + CodeGraph (8 平台)
**目标**: 让 DevSquad 技能可被 Claude Code/Cursor/Codex/OpenCode 等平台使用

**功能需求**:

1. 新增 `HostAdapter` 协议:
   - `format_skill(skill_content: str) -> str`: 将 SKILL.md 格式化为目标平台格式
   - `install_skill(skill_path: Path) -> bool`: 安装技能到目标平台
   - `get_platform() -> str`: 返回平台标识
2. 内置适配器:
   - `TraeHostAdapter`: 当前 Trae 格式（默认）
   - `ClaudeCodeAdapter`: `.claude/` 目录格式

   - `GenericAdapter`: 通用 SKILL.md 格式（`npx skills add` 兼容）
3. SkillRegistry 集成:
   - `register_skill()` 接受可选的 `host` 参数
   - 可将同一技能注册到多个平台

**验收标准**:
- 至少 3 个适配器可正确格式化和安装技能
- Trae 适配器与当前行为完全一致
- GenericAdapter 生成 `npx skills add` 兼容格式

## 8. User Stories

> **格式**: As a [role], I want [feature] so that [benefit]
> **覆盖增强项**: V39-01 ~ V39-07 + RBAC/Audit + 多宿主

### V39-01 代码知识图谱

**US-1**: As a **worker**, I want a persistent CodeKnowledgeGraph so that I don't have to re-scan the codebase with Read/Grep on every task, reducing tool-call waste by ≥30%.

**US-2**: As an **architect**, I want `find_callers(function_name)` and `get_call_graph(entry_point)` queries that respond in <50ms, so that I can reason about blast radius of refactors without manually grepping.

### V39-02 MCP codegraph_explore

**US-3**: As an **external agent** (Claude / GPT), I want to call `codegraph_explore` via MCP, so that I can reuse DevSquad's indexed code knowledge instead of rebuilding it in my own context.

### V39-03 YAGNI 梯子

**US-4**: As a **coder**, I want `YagniChecker` to flag speculative abstractions (single-adapter seams, unused parameters, premature config), so that I follow the YAGNI principle and don't write code that may never be needed.

**US-5**: As a **reviewer**, I want a YAGNI ladder that scores each micro-task on a 0-3 ladder (0=removable, 3=essential), so that I can identify and challenge tasks scoring 0-1 before they consume implementation budget.

### V39-04 三维度 Prompt 调节

**US-6**: As a **PM**, I want `PromptDials` with three independent axes (verbosity / formality / risk-aversion, each 1-5), so that I can tune worker prompts without editing templates — the same skill behaves differently for a brainstorm vs a security audit.

### V39-05 Redesign Audit 第三阶段

**US-7**: As a **reviewer**, I want `TwoStageReviewGate` to add a third "redesign audit" stage that runs `deletion_test` on proposed modules, so that shallow pass-through modules are caught at review time rather than in production.

### V39-06 RBAC/Audit 集成

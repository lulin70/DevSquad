## 一、项目背景与动机

### 1.1 背景

DevSquad V4.0.11 已具备 185+ 核心模块、4603+ 测试、80.03%+ 覆盖率、8.9/10 成熟度。在完成 Matt Pocock .claude skills 评估后（见 [V4.0.11_Matt_Pocock_Skills_Evaluation.md](../audits/V4.0.11_Matt_Pocock_Skills_Evaluation.md)），识别出 21 个技能中 13 个值得借鉴、4 个值得学习。

用户进一步指出 **UI/UX 设计是 DevSquad 比较薄弱的环节**，要求评估 GitHub 上的 taste-skill (Leonxlnx, 28.5K★, MIT) 和 impeccable (Paul Bakaus, 15K★, Apache 2.0) 两个开源 UI/UX Skill，吸收借鉴融合到 DevSquad。UI/UX 评估报告见 [V4.1.0_UIUX_Skills_Evaluation.md](../audits/V4.1.0_UIUX_Skills_Evaluation.md)，识别出 10 项可借鉴/学习（P0×3 + P1×3 + P2×4）。

### 1.2 动机

**不是功能堆砌，而是理念融合**。本次升级的核心目标不是简单地把 Matt 的 18 个技能点 + UI/UX 10 项机械地添加到 DevSquad，而是：

1. **吸收 Matt 数十年工程经验的纪律** — The Pragmatic Programmer / DDD / Extreme Programming / A Philosophy of Software Design
2. **强化 DevSquad 7 角色的工程素养** — 让每个角色都能运用 deep/shallow 词汇、deletion test、red-capable 调试、grilling 访谈等纪律
3. **补全 DevSquad 的追溯链** — GLOSSARY.md 纯术语表 + ADR 架构决策记录体系 + DESIGN.md 设计准则
4. **增强 DevSquad 的 UI/UX 能力** — 补全现有 ~10 条规则到 50+ 条确定性规则，新增 Taste Dials 视觉层，补全 7 design pillars 词汇
5. **保持 DevSquad 7 角色并行协作特色** — 不退化为 Matt 的单 Agent 访谈模式，而是让 7 角色的协作更有纪律

#### GOAL-4: V4.1.0 版本发布

**成功标准**:
- VERSION 文件更新为 4.1.0
- 15 项版本一致性检查全部 PASS
- CHANGELOG / SPEC / SKILL / README 全量同步
- Git commit + push 成功

### 2.2 非目标 (Out of Scope)

| 非目标 | 原因 |
|--------|------|
| 直接调用 Matt 的 .claude skills | Matt 是 Markdown 指令，DevSquad 是 Python 代码，借鉴方法论而非照搬实现 |
| 直接调用 taste-skill / impeccable | 两者是 Markdown 指令型 skill，DevSquad 是 Python 框架，借鉴方法论而非照搬实现 |
| 退化为单 Agent 访谈模式 | DevSquad 7 角色并行协作是核心特色，不退化 |
| 新增外部依赖 | 所有借鉴通过修改现有模块实现，不引入新依赖 |
| 重写现有模块 | 借鉴是增强而非重写，保持向后兼容 |
| DevSquad 自身 UI 重设计 | 本次增强 UI/UX **审查能力**（UIUXAnalyzer 规则库 + TasteDials），不重设计 DevSquad Dashboard 自身 UI |
| 实现具体前端 UI 设计 | DevSquad 不替代前端工程师，不做 minimalist/brutalist 风格预设 |

### 3.1 P0 强借鉴 (7 项) — 优先级最高

#### P0-1: tdd — tautological 检测 + seams-up-front

**借鉴来源**: [tdd/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/tdd/SKILL.md)
**落地模块**: `scripts/collaboration/test_quality_guard.py`
**Matt 理念**:
- red → green (NO refactor) — refactor 属于 review 阶段
- Tautological 反模式: `expect(add(a,b)).toBe(a+b)` 永远通过
- Seams-up-front: 写测试前先确认 seam
- Vertical slice tracer bullet

**DevSquad 融合**:
- `detect_tautological_test()` — AST 分析断言是否重算实现逻辑
- `verify_seams_confirmed()` — 检查测试是否有 seam 确认记录
- Iron Rule 2 强化: 明确 "refactor 属于 review skill (P8)，不属于 TDD loop"
**验收标准**: 新增 ≥15 测试，覆盖 tautological 检测 + seams 确认

#### P0-2: domain-modeling — GLOSSARY.md + ADR 体系

**借鉴来源**: [domain-modeling/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/domain-modeling/SKILL.md)
**落地模块**: `docs/spec/GLOSSARY.md` + `docs/adr/` + `scripts/collaboration/role_skill_loader.py`
**Matt 理念**:
- CONTEXT.md 是纯术语表，禁止实现细节
- ADR 三准则: hard-to-reverse + surprising-without-context + real-tradeoff
- Inline update — 术语确定时立即更新

**DevSquad 融合**:
- 创建 `docs/spec/GLOSSARY.md` — DevSquad 术语表（Module/Seam/Depth/Leverage/Locality/Anchor/Gate/Consensus 等）

#### P0-3: improve-codebase-architecture — deletion test + HTML 报告

**借鉴来源**: [improve-codebase-architecture/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/improve-codebase-architecture/SKILL.md)

#### P0-4: diagnosing-bugs — red-capable gate + [DEBUG-xxx] tag

**借鉴来源**: [diagnosing-bugs/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/diagnosing-bugs/SKILL.md)
**落地模块**: `scripts/collaboration/verification_gate.py` + `scripts/collaboration/execution_guard.py`

**DevSquad 融合**:

#### P0-5: codebase-design — deep/shallow 词汇 + one-adapter-two-adapters

**借鉴来源**: [codebase-design/SKILL.md](https://github.com/mattpocock/skills/blob/main/skills/engineering/codebase-design/SKILL.md)
**落地模块**: `scripts/collaboration/yagni_checker.py` + 架构师 role SKILL.md
**Matt 理念**:
- Deep module = small interface + lots of implementation

- The interface is the test surface.
- Deletion test — 删除后复杂度消失=pass-through

**DevSquad 融合**:
- `yagni_checker.py` 新增 `check_premature_seam(module_path)` — 检测单 adapter 的过早 seam
- 架构师 role SKILL.md 吸收 deep/shallow 词汇

**验收标准**: premature seam 检测 + 架构师 SKILL.md 更新 + ≥10 测试

### 3.4 UI/UX 增强 (10 项) — 补全 DevSquad UI/UX 薄弱环节

> **评估报告**: [V4.1.0_UIUX_Skills_Evaluation.md](../audits/V4.1.0_UIUX_Skills_Evaluation.md)
> **7 角色共识**: 10/10 加权通过（7 项全票 + 3 项 security 弃权）

#### P0-UI-1: 46 确定性规则引擎（impeccable）

**借鉴来源**: [pbakaus/impeccable](https://github.com/pbakaus/impeccable) — 46 deterministic detector rules
**落地模块**: `scripts/qa/uiux_analyzer.py` 新增 `DeterministicRuleEngine` 类
**impeccable 理念**:
- 46 条规则纯确定性检测，不需要 LLM/API key
- 基于 7 design pillars 组织（Typography/Color/Spatial/Responsiveness/Interactions/Motion/UX writing）
- 每条规则有明确的 pass/fail 判定逻辑

**DevSquad 融合**:
- 新增 `DeterministicRuleEngine` 类，管理 46+ 条规则
- 7 pillars 映射到 DevSquad 4 维度（a11y/interaction/layout/ux_antipattern）
- 扩展规则库从 ~10 条到 50+ 条
- 规则格式：`Rule(id, pillar, severity, check_fn, fix_hint)`
**验收标准**: ≥46 条规则 + DeterministicRuleEngine 类 + ≥20 测试

#### P0-UI-2: Taste Dials 视觉版（taste-skill）

**借鉴来源**: [Leonxlnx/taste-skill](https://github.com/Leonxlnx/taste-skill) — Taste Dials
**落地模块**: `scripts/qa/taste_dials.py` 新建（区别于已有的 `scripts/collaboration/prompt_dials.py`）
**taste-skill 理念**:
- 三个 0-1 范围参数控制视觉品味：DESIGN_VARIANCE / MOTION_INTENSITY / VISUAL_DENSITY
- 参数化品味，而非固定规则

**DevSquad 融合**:
- 新建 `TasteDials` 类（0-1 范围，区别于 PromptDials 的 1-5）
- UIUXAnalyzer 审计时读取 TasteDials 配置，动态调整阈值
  - VISUAL_DENSITY 高 → 放宽布局密度检测阈值
  - MOTION_INTENSITY 高 → 放宽动效规则
  - DESIGN_VARIANCE 高 → 放宽一致性检测
- 与 PromptDials 职责分离：PromptDials 管 prompt，TasteDials 管视觉
**验收标准**: TasteDials 类 + UIUXAnalyzer 集成 + ≥12 测试

#### P1-UI-1: Anti-pattern Bans 扩展（taste-skill）

## 八、User Stories

> **格式**: As a [role], I want [feature] so that [benefit]
> **覆盖角色**: Architect / Tester / PM / Coder / Security / DevOps / UI Designer / 所有角色

### Architect 角色故事

**US-1**: As an **architect**, I want to use `deep`/`shallow` module vocabulary when reviewing designs, so that I can communicate architecture decisions precisely instead of relying on vague terms like "God Class" or "complexity".

**US-2**: As an **architect**, I want a `deletion_test(module_path)` tool that detects pass-through modules, so that I can identify shallow modules that add complexity without value and propose deepening opportunities.

### Tester 角色故事

**US-3**: As a **tester**, I want `detect_tautological_test()` to flag tests whose assertions recompute the implementation logic, so that I can catch low-value tests that pass regardless of correctness.

**US-4**: As a **tester**, I want `verify_seams_confirmed()` to require a seam confirmation before writing each test, so that tests are designed around real seams rather than testing implementation details.

### PM 角色故事

**US-5**: As a **PM**, I want `grilling_mode()` to ask one question at a time with recommended answers, so that users are not overwhelmed by multi-question prompts and the design tree is explored branch by branch.

**US-6**: As a **PM**, I want `explore-before-ask` semantics that query the CodeKnowledgeGraph before asking the user, so that I can reduce redundant questions and respect the user's time.

### Coder 角色故事

**US-7**: As a **coder**, I want `check_premature_seam(module_path)` to detect single-adapter "hypothetical seams", so that I don't introduce speculative abstractions before a second adapter justifies the seam.

**US-8**: As a **coder**, I want `[DEBUG-xxx]` tags as the only allowed prefix for debug logging, so that I can clean up all debug code with one grep before merging.

### Security 角色故事

**US-9**: As a **security** reviewer, I want `red-capable gate` to block Phase 2 debugging until a deterministic, fast, agent-runnable red test exists, so that we never debug without a reliable reproducing command.

### DevOps 角色故事

**US-10**: As a **DevOps** engineer, I want `apply_no_op_test(skill_content)` to detect no-op lines in SKILL definitions, so that I can prevent sediment and sprawl in role skill files before they ship.

### UI Designer 角色故事

**US-11**: As a **UI designer**, I want `DeterministicRuleEngine` with 46+ deterministic UI/UX rules (no LLM required), so that I can catch typography/color/spatial/responsiveness regressions deterministically in CI.

**US-12**: As a **UI designer**, I want `TasteDials` (DESIGN_VARIANCE / MOTION_INTENSITY / VISUAL_DENSITY in 0-1 range) to parameterize visual taste, so that UI/UX audits adapt thresholds to the project's visual intent instead of fixed rules.

### 所有角色共享故事

**US-13**: As **any role**, I want `GLOSSARY.md` as a pure terminology table (no implementation details), so that I have a single source of truth for terms like Seam / Deep module / Anchor / Gate / Consensus.

**US-14**: As **any role**, I want `ADR` (Architecture Decision Records) written only when three criteria are met (hard-to-reverse + surprising-without-context + real-tradeoff), so that the decision log stays high-signal and not every minor choice becomes an ADR.

**US-15**: As **any role**, I want `DESIGN.md` to capture project-specific design guidelines (Morandi palette / 4pt grid / motion constraints), so that UI/UX audits evaluate against the project's actual design intent rather than generic best practices.

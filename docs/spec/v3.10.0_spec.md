# DevSquad V3.10.0 规格说明书

> **版本**: 0.9（草案）  
> **日期**: 2026-07-01  
> **目标发布**: V3.10.0  
> **主题**: 引入 ponytail 式最小实现约束与 headroom 式上下文压缩，提升多 Agent 协作质量与成本控制力

---

## 1. 背景与动机

DevSquad V3.9.2 已完成 P0–P3 全量修复，综合评分达到 8.5/A-，硬约束 13/13 通过，测试覆盖率 68.15%。剩余主要技术债集中在：

1. **7-role 并行协作容易过度工程**：角色在生成输出时倾向于添加未请求的抽象、依赖和文件。
2. **上下文压缩缺少内容类型感知**：当前 `ContextCompressor` 以阈值触发 + 截断/摘要为主，对 JSON、代码、日志等不同内容使用同一套逻辑。
3. **跨 Agent 上下文冗余**：每个 Worker 重复读取完整项目上下文，长任务下 token 成本高。
4. **失败经验未形成规则复利**：任务失败/重试的根因未自动沉淀为可注入的规则。

本规格借鉴：
- **ponytail** 的 `AGENTS.md` 行为约束层与「懒惰阶梯」
- **headroom** 的 `ContentRouter` + `SmartCrusher` + `CCR` 可逆压缩架构
- **headroom** 的 `SharedContext` / `headroom learn` 跨 Agent 记忆与失败学习机制

---

## 2. 目标

1. **质量目标**：将综合评分从 8.5/A- 提升至 9.0/A。
2. **覆盖目标**：测试覆盖率从 68.15% 提升至 80%+。
3. **成本目标**：长任务场景下输入 token 降低 30–60%（最终目标 60–90%）。
4. **质量目标**：抑制 7-role 并行中的过度工程，输出代码/文档量减少 20–50%。
5. **学习目标**：建立失败 → 规则提炼 → 自动注入的闭环。

## 4. 用户故事

### US-1：产品经理希望减少无效输出
> 作为 PM，我希望 DevSquad 在生成 PRD/方案时自动抑制未请求的功能和依赖，使输出更聚焦。

验收标准：
- 注入规则后，典型任务输出字数/文件数平均下降 ≥20%
- 不降低需求覆盖度（人工抽检 ≥90% 满足需求）

### US-2：开发者希望长任务不被上下文压垮
> 作为开发者，我希望在分析大型代码库或长日志时，DevSquad 能智能压缩上下文而不丢失关键信息。

验收标准：
- JSON 工具输出类场景 token 降低 ≥60%
- 压缩后 LLM 回答正确率相对基线下降 ≤5%

### US-3：运维希望控制多 Agent 成本
> 作为运维，我希望为每个 dispatch 任务设置 token 预算，超预算时自动告警或降级。

验收标准：
- 支持 `total_input_budget`、`per_role_input_budget`
- 接近预算时触发压缩升级或提前终止
- dashboard/API 暴露预算使用情况

### US-4：团队希望错误不再重复
> 作为团队，我希望 DevSquad 从失败任务中自动学习规则，减少同类错误复现。

验收标准：
- 失败任务自动触发规则提炼
- 高置信度规则注入 `.devsquad.yaml`
- 同类错误复现率下降 ≥30%（基于 benchmark）

## 5. 架构设计

### 5.1 总体架构

```
User Task
    │
    ▼
[InputValidator]
    │
    ▼
[PromptAssembler] ── PonytailRuleInjector ──► 角色 prompt
    │
    ▼
[Coordinator]
    │
    ├── TokenBudget (per dispatch + per role)
    │
    ├──► Worker 1 ──► Scratchpad (CompressedScratchpadEntry)
    ├──► Worker 2 ──► Scratchpad (CompressedScratchpadEntry)
    └──► Worker N ──► Scratchpad (CompressedScratchpadEntry)
    │
    ▼
[ContextCompressor] ── ContentRouter ── SmartCrusher/CodeCompressor
    │
    ▼
[CCRStore] ◄──── devsquad_retrieve tool
    │
    ▼
[ConsensusEngine] ──► Report
```

### 5.2 PonytailRuleInjector

新增 `scripts/collaboration/ponytail_rule_injector.py`：

```python
PONYTAIL_RULES = """
You are a lazy senior developer. Lazy means efficient, not careless.
Before producing output, stop at the first rung that holds:
1. Does this need to be built at all? (YAGNI)
2. Does the standard library already do this? Use it.
3. Does a native platform feature cover it? Use it.
4. Does an already-installed dependency solve it? Use it.
5. Can this be one line? Make it one line.
6. Only then: write the minimum code/doc that works.
Rules:
- No abstractions that weren't explicitly requested.
- No new dependency if it can be avoided.
- Deletion over addition. Boring over clever. Fewest files possible.
- Mark intentional simplifications with a `ponytail:` comment.
Not lazy about: input validation at trust boundaries, error handling that prevents data loss, security, accessibility, anything explicitly requested.
"""
```

注入点：
- `PromptAssembler._build_instruction()` 在 `quality_control.minimal_implementation=True` 时追加
- 默认开启，可通过 `.devsquad.yaml` 关闭

### 5.3 ContentRouter + SmartCrusher

扩展 `ContextCompressor`：

```python
class ContentType(Enum):
    JSON_ARRAY = "json_array"
    CODE = "code"
    LOG = "log"
    PLAIN_TEXT = "plain_text"
    HTML = "html"
    DIFF = "diff"

class ContentRouter:
    def detect(self, text: str) -> ContentType: ...

class SmartCrusher:
    def crush_json_array(self, items: list[dict]) -> str: ...
```

`SmartCrusher` 对 JSON 数组：
- 提取所有项共有的常量字段
- 保留首尾项、异常项、错误项
- 使用字段方差/唯一性选择代表性子集
- 输出：`[1000 items compressed to 20; retrieve full: trace_id=abc]`

### 5.4 CCRStore

新增 `scripts/collaboration/ccr_store.py`：

```python
class CCRStore:
    def store(self, original: str, metadata: dict | None = None) -> str: ...
    def retrieve(self, trace_id: str, query: str | None = None) -> str: ...
    def delete_expired(self, ttl_days: int = 7) -> int: ...
```

- 后端：SQLite（默认 `data/ccr_store.db`）
- 缓存：LRU（内存）+ TTL（数据库）
- 检索：可选 BM25 局部检索

### 5.5 TokenBudget

在 `scripts/collaboration/models.py` 新增：

```python
@dataclass
class TokenBudget:
    total_input_budget: int = 100_000
    per_role_input_budget: int = 20_000
    output_budget: int = 10_000
    warning_ratio: float = 0.8
```

`Coordinator` 在调度每个 Worker 前检查预算，接近阈值时：
1. 提升压缩级别
2. 缩短 briefing
3. 必要时跳过非关键角色

### 5.6 CompressedScratchpad

`Scratchpad` 支持写入压缩条目：

```python
class CompressedScratchpadEntry:
    summary: str
    trace_id: str
    original_size: int
    compressed_size: int
```

Worker 读取时默认拿到 summary；当输出中包含 `devsquad_retrieve(trace_id=..., query=...)` 时，Coordinator 自动取回原文并注入。

### 5.7 失败学习闭环

扩展 `RetrospectiveSkill`：

```python
class LearnedRule:
    rule_text: str
    trigger_condition: str
    confidence: float
    source_task_id: str
    created_at: datetime
```

流程：
1. 任务失败/重试/上下文超限/共识未达成 → 触发 RetrospectiveSkill
2. 提炼 `LearnedRule`
3. 置信度 ≥0.8 → 写入 `.devsquad.yaml` `quality_control.learned_rules`
4. 置信度 0.5–0.8 → 进入 `data/tier2/corrections.json` 候选池
5. `PromptAssembler` 加载 `.devsquad.yaml` 时自动注入 learned rules

## 6. 接口变更

### 6.1 `.devsquad.yaml`

新增配置段：

```yaml
quality_control:
  enabled: true
  minimal_implementation: true
  ponytail_markers: true
  learned_rules:
    - rule: "Always prefer pathlib over os.path when available"
      trigger: "file_path_manipulation"
      confidence: 0.85

context_compression:
  enabled: true
  default_level: "smart"  # none | snip | session_memory | full_compact | smart
  content_router:
    json_array_crush: true
    code_ast_compress: false  # P2
    log_compress: true
  ccr:
    enabled: true
    db_path: "data/ccr_store.db"
    ttl_days: 7

token_budget:
  total_input_budget: 100000
  per_role_input_budget: 20000
  output_budget: 10000
  warning_ratio: 0.8
```

### 6.2 Python API

```python
# TokenBudget 透传
result = disp.dispatch(
    "Analyze large codebase",
    token_budget=TokenBudget(total_input_budget=50_000),
)

# 强制最小实现规则关闭
result = disp.dispatch(
    "Generate comprehensive architecture doc",
    quality_control={"minimal_implementation": False},
)
```

### 6.3 CLI

```bash
# 启用 smart 压缩
python scripts/cli.py dispatch -t "Analyze logs" --compression smart

# 设置 token 预算
python scripts/cli.py dispatch -t "Design API" --input-budget 50000
```

## 8. E2E 验证要求

每个 Phase 结束前必须完成模拟真实用户的端到端测试：

1. **输入**：使用真实项目代码库作为输入
2. **任务类型**：覆盖简单 / 中等 / 复杂三类任务
3. **验证项**：
   - 输出是否满足需求
   - 是否引入未请求依赖
   - 是否在 token 预算内完成
   - 压缩后关键信息是否可召回
4. **失败学习**：验证同类错误复现率是否下降

## 9. 风险与成本评估

| 建议项 | 主要风险 | 缓解措施 | 估算成本 |
|--------|----------|----------|----------|
| Phase 1 Ponytail 规则注入 | 规则过于激进导致必要抽象被砍掉 | 保持规则简短；明确「不懒」底线；A/B 测试 | 低（1–2 人周） |
| Phase 2 ContentRouter + SmartCrusher | JSON 采样丢失关键异常项 | 异常/错误项无条件保留；短内容跳过压缩 | 中（2–4 人周） |
| Phase 3 CCR 可逆压缩 | 存储膨胀；retrieve 调用增加交互轮次 | SQLite 加 TTL/LRU；设置 retrieve 预算 | 中（3–5 人周） |
| Phase 3 Token 预算 | 预算设置不当导致输出被过早截断 | 默认宽松；分阶段收紧；提供 override | 中（2–3 人周） |
| Phase 4 失败学习闭环 | 垃圾规则积累；规则冲突 | 置信度门槛；人工审核高影响规则 | 中高（4–6 人周） |

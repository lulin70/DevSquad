# DevSquad V4.0.0 规格说明书

> **版本**: 1.0（草案→共识）
> **日期**: 2026-07-07
> **目标发布**: V4.0.0
> **主题**: 借鉴上游 TraeMultiAgentSkill v2.7 的 6 个特性，引入 Loop Engineering 五步闭环、UI/UX 巡检视觉回归、对抗验证、DAG 可视化、Autonomous 自主迭代、插件热加载，从"一次性派发"升级为"循环直到完成"的自适应协作平台

## 1. 背景与动机

### 1.1 上游仓库分析

DevSquad V3.9.3 已完成 V3.10.0 全部 4 个 Phase，综合评分 8.6/A-，硬约束 13/13 通过。通过对上游 [TraeMultiAgentSkill v2.7](https://github.com/weiransoft/TraeMultiAgentSkill)（88 commits，最新 2026-07-01）的深度分析，识别出 6 个值得借鉴的特性。

### 1.2 DevSquad 当前局限

| 局限 | 根因 | 影响 |
|------|------|------|
| **一次性派发** | dispatch 是 pre→exec→post 单向流水线 | 任务未完成时需人工重新触发 |
| **无独立验证** | ConsensusEngine 是前置共识门，缺少后置独立评估 | 产出质量依赖共识而非独立验证 |
| **无视觉质量门禁** | 26 个 UI E2E 测试只验证功能，不检查视觉质量 | A11y/布局问题无法自动发现 |
| **单一协作模式** | 只有"派发→执行→共识"一种模式 | 无法应对对抗验证/锦标赛等场景 |
| **无生命周期可视化** | lifecycle_protocol 有 11 阶段但无图形展示 | 用户难以理解阶段依赖 |
| **静态组件注册** | dispatch_component_factory 是静态的 | 添加组件需改代码重启 |

### 1.3 借鉴来源

| 特性 | 上游文件 | 上游行数 | 借鉴思路 |
|------|---------|---------|---------|
| Loop Engineering | `scripts/loop_engineering/` (12 文件) | ~3500 行 | Discovery→Handoff→Verification→Persistence→Scheduling |
| UI/UX 巡检 | `scripts/uiux_analyzer.py` | ~400 行 | 4 大检测维度 + Playwright 单次探针 |
| 视觉回归 | `scripts/visual_regression.py` | ~350 行 | PIL ImageChops 像素级 Diff + SSIM |
| 对抗验证 | `scripts/dynamic_workflow/` | ~600 行 | adversarial-verify 模式 |
| DAG 可视化 | `scripts/dag_visualizer.py` | ~300 行 | Mermaid/JSON/DOT 三种格式 |
| Autonomous | `scripts/autonomous/` (9 组件) | ~2000 行 | plan→dev→verify→fix 4 阶段循环 |
| 插件热加载 | `scripts/plugins/` + hot_reload_watcher | ~500 行 | drop-in 目录 + mtime 轮询 |

### 1.4 DevSquad 独有优势（保持不变）

- 三贤者并行投票架构（asyncio.gather）
- ConsensusEngine 前置共识门（HC-2）
- CCRStore 可逆压缩（SQLite+LRU+TTL）
- TokenBudget 预算管理

- 企业级 RBAC + SQLite 审计
- Streamlit Dashboard + FastAPI 服务化

## 4. 用户故事

### US-1：开发者希望任务自动循环直到完成
> 作为开发者，我希望 DevSquad 能自动循环执行任务，直到目标达成或触发上限，无需人工重新触发。

验收标准：
- 支持 `loop_mode=True` 启用循环派发
- 每轮自动 Discovery → Handoff → Verification → Persistence → Scheduling
- 独立 Evaluator 验证产出（STRICT 模式）
- 达成目标自动停止，或触发 max_iterations 上限
- 人类检查点每 N 轮触发（可配）

### US-2：测试专家希望自动检测 UI 视觉问题
> 作为测试专家，我希望 DevSquad 能自动检测 Dashboard 的 A11y 问题、布局错误和视觉回归。

验收标准：
- 4 大检测维度：可访问性/交互质量/布局响应式/UX 反模式
- 像素级 Diff 检测视觉回归
- 集成到 UI E2E 测试套件
- 生成结构化问题报告（severity/category/fix）

### US-3：架构师希望对抗验证关键决策
> 作为架构师，我希望对关键架构决策进行对抗性验证，红方挑战蓝方方案，提高决策鲁棒性。

验收标准：
- 支持 `adversarial_verify` 模式
- 红方（挑战者）生成反例和边界 case
- 蓝方（防御者）响应并加固方案
- 裁判评估胜负并输出改进建议
- 与 ConsensusEngine 整合，对抗结果影响共识决策

### US-4：运维希望可视化生命周期依赖
> 作为运维，我希望在 Dashboard 看到 11 阶段生命周期的 DAG 依赖图，理解阶段关系。

验收标准：
- Mermaid 格式渲染 DAG 图
- 支持 JSON/DOT 格式导出
- 集成到 Dashboard "Phases" 页面
- 节点状态实时更新（pending/running/completed/failed）

### US-5：PM 希望团队自主完成迭代
> 作为 PM，我希望设置目标后，DevSquad 团队自主 plan→dev→verify→fix 循环，直到迭代完成。

验收标准：
- 支持 `autonomous_mode=True` 启用自主迭代
- 4 阶段循环：plan→dev→verify→fix
- 断点续跑（SHA256 校验 + 备份恢复）
- 智能确认三态决策（smart/whitelist-only/blacklist-only）
- 与 ConsensusEngine 协调（自主模式不绕过共识门）

### US-6：开发者希望运行时扩展组件
> 作为开发者，我希望通过 drop-in 目录添加新组件，无需修改核心代码或重启服务。

验收标准：
- 支持 `plugins_extra/*.py` drop-in 目录
- 运行时 mtime 轮询自动发现新组件
- Hot Register API（`hot_register` / `hot_unregister`）
- reload 失败回滚到旧实例

```python
class LoopKernel:
    """Loop Engineering 五步闭环编排核心。

    Discovery → Handoff → Verification → Persistence → Scheduling
    """

    def run(self, objective: str) -> LoopRunReport:
        while not self._stop_requested:
            cycle = self.run_one_cycle(objective, iter_index)
            # 1. Discovery: 发现本轮该做什么
            # 2. Handoff: 生成工作项 + 调用 Worker
            # 3. Verification: 独立 Evaluator 评估
            # 4. Persistence: 写入 UnifiedMemory
            # 5. Scheduling: 决策下一步 (CONTINUE/FIX/HUMAN_CHECKPOINT/STOP)

            if decision.action == SchedulingAction.STOP_SUCCESS:
                break
            if decision.action == SchedulingAction.STOP_FAILURE:
                break
            if iter_index >= self._config.max_iterations:
                break
```

```python
class AutonomousLoopController:
    def run(self, objective: str) -> RunReport:
        while not self._should_stop():
            # plan: 分解任务 + 角色匹配
            plan = self._plan(objective)
            # dev: 执行开发
            result = self._dev(plan)
            # verify: 验证结果（含 ConsensusEngine 共识门）
            verified = self._verify(result)
            if verified.success:
                break
            # fix: 基于验证结果修复
            self._fix(verified.errors)
```

#### 5.6.3 与 ConsensusEngine 协调

**关键约束**：Autonomous 模式不绕过 ConsensusEngine 前置共识门（HC-2）。

```python
def _verify(self, result):
    # 1. ConsensusEngine 前置共识门检查
    gate_result = self._consensus_engine.check_gate(result)
    if not gate_result.approved:
        return VerifiedResult(success=False, errors=[gate_result.reason])
    # 2. 独立 Evaluator 验证
    return self._evaluator.evaluate(result)
```

### 7.1 分阶段实施

| 阶段 | 特性 | 工作量 | 依赖 |
|------|------|--------|------|

3. 集成测试（接入 dispatch pipeline）
4. E2E 测试（≥10 个场景）
5. 文档更新（CHANGELOG + README + SKILL.md）

## 11. 团队全员共识记录

### 11.1 共识评审流程

7 角色（架构师/PM/安全/测试/开发/DevOps/UI）对本 spec 进行评审，各角色关注点如下：

### 11.2 架构师评审

**关注点**：架构一致性、模块边界、技术债
**意见**：
- ✅ Loop Engineering 的 Protocol 组合模式与 DevSquad 现有 mixin 架构一致
- ✅ 对抗验证扩展 ConsensusEngine 而非新建模块，符合 Surgical Changes 原则
- ⚠️ Autonomous 模式与 Loop Engineering 有功能重叠，需明确边界
- **决议**：Autonomous 复用 LoopKernel，不重复实现循环逻辑

### 11.3 PM 评审

**关注点**：用户价值、优先级、交付节奏
**意见**：
- ✅ P1 优先级正确：Loop Engineering 是核心能力升级
- ✅ US-1（循环直到完成）是最高用户价值
- ⚠️ 7.5-12.5 天工时偏长，建议 P2-2（DAG）和 P3-2（热加载）可并行
- **决议**：P1→P2 串行，P3 并行实施

### 11.4 安全专家评审

**关注点**：安全边界、路径穿越、权限提升
**意见**：
- 🔴 插件热加载必须安全审查（路径穿越三层防护）
- 🔴 Autonomous 模式的 git_driver 需要 RBAC 权限检查
- ✅ 对抗验证不引入新攻击面
- **决议**：P3-2 插件热加载必须通过安全审查才能发布

### 11.5 测试专家评审

**关注点**：测试覆盖、E2E 充分性、无幽灵功能
**意见**：
- ✅ 每特性 ≥10 个 E2E 测试，总 ≥60 个
- ✅ 幽灵功能检查：每个特性必须接入 dispatch pipeline
- ⚠️ Loop Engineering 的无限循环场景需专项测试
- **决议**：新增 `test_loop_safety_limits` 专项测试类

### 11.6 开发者评审

**关注点**：代码简洁、可维护性、YAGNI
**意见**：
- ✅ 借鉴上游但不照搬，保持 DevSquad 风格
- ✅ 软依赖策略（Pillow/numpy/playwright）符合 YAGNI
- ⚠️ Autonomous 9 组件过多，建议精简为 5 个
- **决议**：Autonomous 精简为 loop_controller + run_state + notes_memory + smart_confirmation + git_driver

### 11.7 DevOps 评审

**关注点**：CI/CD、部署、监控
**意见**：
- ✅ 新增 API 端点需要健康检查
- ✅ DAG 可视化需要缓存策略
- ⚠️ 插件热加载需要审计日志持久化
- **决议**：热加载审计日志写入 SQLite audit 表

### 11.8 UI 设计师评审

**关注点**：Dashboard UX、视觉一致性
**意见**：
- ✅ DAG View 使用 Streamlit 原生 `st.mermaid()`，无需额外依赖
- ✅ QA 页面展示巡检结果，使用 Morandi 配色
- ⚠️ Dashboard sidebar 已有 6 个选项，新增 2 个需评估导航体验
- **决议**：sidebar 新增 "DAG" 和 "QA" 两个选项，总计 8 个

### 11.9 共识结论

**全员一致通过 V4.0.0 spec，7 角色无否决意见。**

关键共识：
1. P1 优先实施 Loop Engineering + UI/UX 巡检
2. Autonomous 复用 LoopKernel，精简为 5 组件
3. 插件热加载必须通过安全审查
4. 每特性 ≥10 E2E 测试，无幽灵功能
5. 版本号 3.9.3 → 4.0.0

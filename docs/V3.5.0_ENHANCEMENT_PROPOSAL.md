# DevSquad V3.5.0 增强方案

> 版本: V2.1 | 日期: 2026-05-05 | 状态: 待审批
> 遵循项目全生命周期规范：文档先行
> 编制依据：上游优点全量盘点 + 现有功能重叠分析 + 7角色共识投票

---

## 一、方案编制方法

### 1.1 编制流程

```
上游优点全量盘点 → 现有功能逐项对照 → 重叠/差距分类 → 增强候选提炼 → 7角色投票 → 共识方案
```

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **不重复造轮子** | 我们已有且更优的功能，不重复添加 |
| **不粗暴堆砌** | 每个增强项必须说明与现有功能的关系（互补/强化/替代），杜绝功能重复 |
| **补短板优先** | 上游有而我们确实缺失的，评估后纳入 |
| **强化优势** | 在我们已有优势的方向上继续深化，而非另起炉灶 |
| **7角色共识** | 所有增强项由团队7角色投票决定，权重加权 |
| **文档先行** | 方案文档审批通过后才开始实现 |

### 3.4 部分重叠项深入分析

这是最关键的分析——避免功能重复，明确互补关系。

#### A6 规范工具链 vs LifecycleProtocol

| 维度 | A6 规范工具链 | LifecycleProtocol | 关系 |
|------|-------------|-------------------|------|
| 定位 | 独立spec操作命令 | 生命周期阶段管理 | **互补** |
| 功能 | spec init/analyze/validate | spec/plan/build/test/review/ship | **LifecycleProtocol已覆盖核心** |
| 差距 | 无 | 缺少spec独立操作能力 | **可强化** |

**结论**：LifecycleProtocol已覆盖核心功能，spec独立操作是锦上添花，优先级低。

#### A8 项目理解生成器 vs AgentBriefing

| 维度 | A8 项目理解生成器 | AgentBriefing | 关系 |
|------|----------------|--------------|------|
| 定位 | 独立脚本，生成项目理解文档 | 调度前上下文注入 | **互补** |
| 输出 | Markdown文档（可独立查看） | 上下文片段（注入提示词） | **不同用途** |
| 触发 | 手动执行 | 自动触发 | **不同场景** |

**结论**：功能角度不同，AgentBriefing缺"项目级理解文档生成"能力，可强化。

#### A16 多角色代码走读 vs 五轴引擎

| 维度 | A16 多角色代码走读 | 五轴引擎 | 关系 |
|------|------------------|---------|------|
| 定位 | 角色走读内容（看什么、怎么评） | 评审共识机制（怎么投票） | **互补** |
| 角色 | 5角色视角定义 | 5轴权重定义 | **我们有引擎缺内容** |
| 输出 | 各角色走读报告 | 共识裁决结果 | **互补** |

**结论**：五轴引擎是"投票机制"，代码走读增强包是"投票内容"，互补不重叠。

#### A21 行为准则体系 vs anti_rationalization.py

| 维度 | A21 行为准则 | anti_rationalization | 关系 |
|------|------------|---------------------|------|
| 定位 | 角色应该做什么（正面指引） | 角色不该找什么借口（反面警示） | **互补** |
| 机制 | 行为规范 + 约束条件 | 借口-反驳对 | **不同策略** |
| 覆盖 | 全角色行为规范 | 50条借口反驳 | **不同维度** |

**结论**：正面指引 vs 反面警示，互补不重叠。但行为准则可融入增强包，无需独立模块。

#### A23 角色专属应用指南 vs concern_pack_loader

| 维度 | A23 角色专属指南 | concern_pack_loader | 关系 |
|------|----------------|---------------------|------|
| 定位 | 每个角色的最佳实践 | 领域知识增强包 | **互补** |
| 触发 | 始终生效 | 关键词匹配激活 | **不同粒度** |
| 内容 | 通用最佳实践 | 领域特定知识 | **不同层次** |

**结论**：角色专属指南是"通用层"，增强包是"领域层"，互补。走读场景的专属指南可纳入代码走读增强包。

#### A25 Vibe Coding 14条规则 vs CLAUDE.md

| 维度 | A25 Vibe Coding | CLAUDE.md | 关系 |
|------|----------------|----------|------|
| 定位 | AI编码行为14条规范 | 项目级AI协作规则 | **部分重叠** |
| 覆盖 | 14条具体规则 | 元规则+测试铁律+交付流程 | **CLAUDE.md更广** |
| 差距 | 无 | 缺少编码行为的具体约束 | **可强化** |

**结论**：CLAUDE.md覆盖更广但缺编码行为具体约束。Vibe Coding核心规则可融入Karpathy增强包，避免独立模块。

## 四、增强候选提炼

### 4.1 提炼原则

从上述分析中提炼增强候选时，严格遵循：
1. **不重复**：与现有功能重叠的，说明互补关系而非重复
2. **可组合**：优先复用现有架构（增强包、五轴引擎、AgentBriefing）
3. **有依据**：每个候选项必须追溯到上游优点编号

| 候选ID | 增强项 | 来源 | 描述 | 与现有功能关系 | 预计工作量 |

| **E2** | 文档一致性检查 | A20 | 代码走读时检查文档与代码差异，明确责任角色 | **新增**，无重叠 | 0.5天 |

### 4.3 功能重复风险检查

| 候选ID | 潜在重复风险 | 消除措施 |
|--------|------------|---------|
| E1 | 与concern_pack_loader角色增强重复 | E1走读增强包是**场景特定**（代码走读），现有3个增强包是**领域特定**（权限/API/数据），层次不同 |
| E2 | 与verification_gate重复 | E2检查**文档vs代码**一致性，verification_gate检查**完成声明vs证据**，维度不同 |
| E3 | 与anti_rationalization重复 | E3管**LLM犯蠢**（不思考就写、过度设计），anti_rationalization管**LLM找借口**（跳过步骤），策略不同 |
| E3 | 与CLAUDE.md规则重复 | E3是**增强包**（关键词激活），CLAUDE.md是**项目规则**（始终生效），粒度不同 |
| E6 | 与code_map_generator重复 | E6生成**项目理解文档**（技术栈+模块+组件），code_map_generator生成**代码结构图**（类+函数+依赖），输出不同 |
| E8 | 与原五轴引擎重复 | E8是**独立工厂函数**create_walkthrough_engine()，不修改原引擎，场景隔离 |

**结论**：所有候选项与现有功能互补不重复，重复风险已通过设计消除。

## 六、文档一致性检查责任定岗

### 6.1 问题定义

代码走读时，文档内容的一致性由谁检查？这是用户明确要求定下来的事项。

### 6.2 责任矩阵

**核心原则：谁最懂那类文档，谁负责那类文档与代码的一致性检查。**

| 文档类型 | 一致性检查主责 | 辅责 | 检查内容 | 严重程度 |
|---------|--------------|------|---------|---------|
| API文档 vs 代码 | 🏗️ 架构师 | 💻 开发者 | 接口签名、参数类型、返回值、错误码是否与文档一致 | 🔴 Critical |
| 安全策略文档 vs 代码 | 🔒 安全专家 | — | 安全配置、权限规则、加密策略是否与文档一致 | 🔴 Critical |
| 测试文档 vs 代码 | 🧪 测试专家 | — | 测试用例、覆盖率目标、测试策略是否与文档一致 | 🟠 High |
| 需求文档 vs 代码 | 📋 产品经理 | — | 功能实现是否与PRD一致、验收标准是否达成 | 🔴 Critical |
| 运维文档 vs 代码 | ⚙️ 运维专家 | — | 部署配置、监控指标、告警阈值、SLA是否与文档一致 | 🟠 High |
| UI设计文档 vs 代码 | 🎨 UI设计师 | — | 交互规范、组件库、视觉规范是否与设计稿一致 | 🟡 Medium |
| 配置文档 vs 代码 | ⚙️ 运维专家 | 🏗️ 架构师 | 环境变量、配置项、默认值是否与文档一致 | 🟡 Medium |
| 变更日志 vs 实际变更 | 📋 产品经理 | 💻 开发者 | CHANGELOG是否完整记录了所有用户可见变更 | 🟡 Medium |
| 版本号一致性 | 🏗️ 架构师 | ⚙️ 运维专家 | _version.py / pyproject.toml / CHANGELOG / README / INSTALL / GUIDE / SKILL.md / skill-manifest.yaml 中版本号是否统一 | 🔴 Critical |

### 6.3 检查触发时机

| 时机 | 触发条件 | 检查范围 |
|------|---------|---------|
| 代码走读时 | 触发code-walkthrough增强包 | 全部9类文档 |
| 生命周期P8(实现)阶段 | 代码提交评审 | API文档、需求文档、测试文档 |
| 生命周期P9(测试)阶段 | 测试评审 | 测试文档、安全策略文档 |
| 生命周期P10(部署)阶段 | 部署评审 | 运维文档、配置文档、变更日志 |
| 版本发布前 | 版本号更新时 | 变更日志、API文档、**版本号一致性** |

### 6.4 严重程度与处置

| 严重程度 | 定义 | 处置 |
|---------|------|------|
| 🔴 Critical | 文档与代码严重不一致，可能导致生产事故 | **阻断发布**，必须修复后才能继续 |
| 🟠 High | 文档与代码不一致，可能影响运维或测试 | **必须记录**，本版本内修复 |
| 🟡 Medium | 文档与代码轻微不一致，不影响功能 | **建议修复**，可推迟到下版本 |

### 6.5 实现方式

文档一致性检查**不独立成包**，作为代码走读增强包（E1）的一个内置维度。每个角色的走读检查清单中自动包含其负责的文档一致性检查项。

### E8: 走读专用五轴引擎

**文件**: 扩展 `five_axis_consensus.py`

**新增工厂函数**: `create_walkthrough_engine()`

**五轴调整**：

| 轴 | 原权重 | 新权重 | 主责角色 | 说明 |
|----|--------|--------|---------|------|
| Correctness | 0.30 | 0.25 | 开发者 + 测试专家 | 逻辑正确、无Bug |
| Security | 0.25 | 0.25 | 安全专家 | 漏洞、合规（保留严格模式否决权） |
| Architecture | 0.20 | 0.20 | 架构师 | 设计模式、模块化 |
| **Operability** | — | **0.15** | 运维专家 | 部署、监控、容灾（替代Performance） |
| Readability | 0.10 | 0.15 | 开发者 + 全角色 | 代码清晰、可维护 |

**为什么用Operability替代Performance**：
- Performance是Architecture + Operability的子集
- Operability更全面：部署可行性 + 日志规范 + 监控埋点 + 容灾能力 + 配置管理 + 性能运维
- 运维专家在原五轴中没有明确位置，Operability轴给予运维专家正式权重

**Operability轴的检查项**：
- 部署可行性（Docker/K8s配置是否完整）
- 日志规范（关键操作是否有日志、日志级别是否合理）
- 监控埋点（核心指标是否有监控、告警阈值是否设置）
- 容灾能力（是否有降级方案、是否有回滚机制）
- 配置管理（配置是否外置、是否有环境隔离）
- 性能运维（资源使用、响应时间、容量规划、SLA）

**与原引擎的关系**（防重复）：
- `create_default_engine()` 保持不变（通用评审场景）
- `create_walkthrough_engine()` 新增（代码走读专用场景）
- 两个引擎独立共存，不互相影响

### 8.3 E4 代码地图多语言支持（后期阶段详细方案）

**实现方式**: 定义 LanguageParser Protocol，提取 PythonParser，新增 JavaScriptParser + GoParser

**架构设计**：

```
LanguageParser Protocol (protocols.py)
├── PythonParser    — 基于ast标准库（从现有代码提取）
├── JavaScriptParser — 基于正则表达式（零外部依赖）
└── GoParser        — 基于正则表达式（零外部依赖）

CodeMapGenerator
├── register_parser(parser)  — 注册语言解析器
├── generate_map(languages=["python","js","go"])  — 按语言过滤
└── get_dependency_graph()   — 多语言依赖图
```

**关键决策**：
- JS/Go使用正则表达式而非tree-sitter，避免引入重量级依赖
- 正则覆盖80%常见结构（类、函数、import），解析失败返回None（优雅降级）
- 默认注册PythonParser，向后兼容，现有调用方无需修改

**文件变更**：
- `protocols.py` 新增 `LanguageParser` Protocol
- `code_map_generator.py` 重构：提取PythonParser，新增register_parser
- 新文件 `language_parsers.py`：PythonParser + JavaScriptParser + GoParser
- `null_providers.py` 新增 `NullLanguageParser`

### 8.4 E5 规范工具链增强（后期阶段详细方案）

**实现方式**: 扩展 LifecycleProtocol + CLI子命令

**新增3个CLI命令**：
- `devsquad spec-init --template requirements|architecture|technical` — 从模板初始化规范文档
- `devsquad spec-analyze --source-dir .` — 分析代码生成规范草稿（复用E4多语言CodeMapGenerator）
- `devsquad spec-validate --template requirements` — 验证规范完整性

**架构设计**：

```
LifecycleProtocol
├── init_spec(template_id, project_info)     — 初始化规范模板
├── analyze_spec(source_dir, template_id)    — 分析代码生成规范
└── validate_spec(spec_path, template_id)    — 验证规范完整性

SPEC_TEMPLATES (Python dataclass)
├── requirements — 需求规格模板（对应P1）
├── architecture — 架构规格模板（对应P2）
└── technical    — 技术规格模板（对应P3）
```

**关键决策**：
- spec子命令用独立命令（spec-init）而非嵌套子命令组（spec init），与现有6命令风格一致
- 模板初始用Python dataclass常量，V3.6.0可迁移到YAML文件
- analyze_spec复用E4的CodeMapGenerator，E4+E5协同

**文件变更**：
- `lifecycle_protocol.py` 新增3个抽象方法 + SpecTemplate + SPEC_TEMPLATES + VIEW_MAPPINGS扩展
- `cli.py` 新增spec-init/spec-analyze/spec-validate子命令
- `api/routes/lifecycle.py` 新增3个REST API端点

## 十、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |

## 一、核心优化内容

### 1.1 新增自动继续机制

#### 问题背景
当任务复杂度超过模型单次思考能力时（如深度分析、大规模代码重构、多文件协同修改），模型可能达到思考次数限制而中断，导致任务未完成。

#### 解决方案

**1. 断点续传设计**
- **触发条件**: 
  - 模型输出包含"篇幅限制"、"思考次数已达上限"等中断提示
  - 任务列表仍有未完成项（pending/in_progress）
  - 代码修改未完成（缺少验证、测试等步骤）

- **续传流程**:
  ```
  Step 1: 自动识别中断点
    - 最后一个成功执行的操作
    - 未完成的任务项
    - 已修改但未验证的代码
  
  Step 2: 自动恢复上下文
    - 重新加载任务列表和当前状态
    - 读取已修改文件的当前内容
    - 获取最近的代码审查/测试结果
  
  Step 3: 自动继续执行
    - 从中断点继续下一个操作
    - 无需用户重复描述需求
    - 保持任务连贯性和一致性
  ```

**2. 分块处理策略**
- 大任务主动分解（示例：重构 20 个文件 → 4 个分块）
- 每块独立验证（完成 → 验证 → 提交 → 继续）
- 失败自动修复（验证失败 → 修复 → 重验）

**3. 进度持久化**
- 存储位置：`.trae-multi-agent/progress.md`
- 记录内容：任务快照、代码变更、验证结果
- 支持操作：自动更新、手动回滚、进度导出

**4. 智能续期策略**
- 续期判断：剩余思考次数 < 2 次时评估
- 续期执行：保存上下文 → 输出提示 → 自动继续
- 用户无感知：无需手动干预

**5. 用户无感知体验**
```
用户看到的:
"正在重构用户模块... (进度 70%)"
↓ （思考次数超限）
"继续执行中... (进度 75%)"
↓ （自动继续）
"重构完成！所有测试通过 ✅"

用户不需要的:
❌ 手动重复需求
❌ 手动检查进度
❌ 手动触发继续
❌ 担心任务中断

系统自动做的:
✅ 自动保存进度
✅ 自动恢复上下文
✅ 自动继续执行
✅ 自动验证结果
```

### 1.2 完善任务管理机制

#### 任务分解规则
**何时使用 todo_write**:
```python
if 任务步骤 >= 3:
    必须使用 todo_write
elif 需要多角色协作:
    必须使用 todo_write
elif 任务复杂度 > 阈值:
    必须使用 todo_write
```

#### 任务状态流转
```
pending → in_progress → completed
         ↓
      blocked (遇到阻塞时)
```

#### 完成标准
- 任务列表所有任务完成才算当前对话完成
- 只有满足以下条件才能标记任务为 completed：
  - 相关子任务完成
  - 相关测试通过
  - 代码审查通过（如适用）

### 1.3 状态更新协议

**每次工具调用后必须输出**（1-3 句话）:
- 已完成的操作
- 即将进行的操作
- 遇到的阻塞/风险（如有）

**示例**:
```
"已完成用户认证模块的代码审查，发现 3 个潜在安全问题。
接下来将生成详细的审查报告并@开发者进行修复。"
```

### 5.1 基础用法

#### 单角色任务
```bash

python3 scripts/trae_agent_dispatch.py \
    --task "设计系统架构：包括模块划分、技术选型、部署方案"

# 明确指定角色
python3 scripts/trae_agent_dispatch.py \
    --task "实现广告拦截功能" \
    --agent solo_coder
```

#### 多角色共识
```bash
# 需求评审（多角色参与）
python3 scripts/trae_agent_dispatch.py \
    --task "需求评审：评估广告拦截功能的可行性和工作量" \
    --consensus true \
    --agents architect,product_manager,test_expert,solo_coder
```

#### 完整项目流程
```bash
# 启动完整项目
python3 scripts/trae_agent_dispatch.py \
    --task "启动项目：安全浏览器广告拦截功能" \
    --project-full-lifecycle

# 1. product_manager: 定义需求
# 2. architect: 设计架构
# 3. test_expert: 制定测试策略

# 5. solo_coder: 功能开发
```

## 七、质量保障措施

### 7.1 代码验证强制要求

**所有代码修改后必须**:
1. 使用 `get_problems` 检查编译错误
2. 运行相关单元测试
3. 如果有错误立即修复并重新验证
4. 直到所有检查通过才能标记任务完成

**验证流程**:
```
代码修改 → get_problems → 有错误？
                              ↓ 是
                         修复错误
                              ↓
                              ↓ 否
                         运行测试 → 失败？
                              ↓ 是
                         修复测试
                              ↓
                              ↓ 否
                         标记完成 ✅
```

### 7.2 测试覆盖率要求

**新增功能必须**:
- 单元测试覆盖率 > 80%
- 分支覆盖率 > 70%
- 所有测试通过
- 测试执行时间 < 1 分钟

**测试生成策略**:
1. 正常场景测试（至少 3 个）
2. 异常场景测试（至少 2 个）
3. 边界条件测试（至少 2 个）
4. 性能测试（至少 1 个）

### 7.3 代码审查流程

**多角色审查**:
- **架构师**: 架构合规性、设计模式使用
- **测试专家**: 测试覆盖率、边界条件处理
- **Solo Coder**: 代码质量、编码规范

**审查清单**:
- [ ] 代码符合架构设计
- [ ] 错误处理完整
- [ ] 日志记录完整
- [ ] 配置可配
- [ ] 测试覆盖全面
- [ ] 性能达标

# 阶段门禁模板 (Stage Gate Template)

## 概述

阶段门禁是全生命周期开发流程中的质量检查点。每个阶段必须通过门禁检查才能进入下一阶段。门禁确保"文档先行、质量优先、不可跳步"。

## 门禁检查流程

```
阶段 N 完成
    ↓
执行门禁检查
    ↓
┌─ 全部通过 → 进入阶段 N+1
├─ 部分通过 → 修复后重检
└─ 不通过 → 返回阶段 N 重新执行
```

## 门禁等级

### 🔴 硬性门禁（必须通过）
- 上游交付物完整性
- 零容忍清单无违规
- 评审通过签字

### 🟡 建议门禁（建议通过）
- 质量指标达标
- 文档完整性
- 最佳实践遵循

### 🟢 信息门禁（仅记录）
- 进度偏差说明
- 风险记录
- 经验教训

## 门禁不通过处理

1. **记录不通过原因**：明确哪些检查项未通过
2. **制定修复计划**：列出需要修复的问题和预计时间
3. **执行修复**：返回对应阶段修复
4. **重新检查**：修复后重新执行门禁检查
5. **升级决策**：如果多次不通过，升级到多角色共识决策

## 8 阶段门禁速查

| 阶段 | 关键门禁 | 硬性检查项 |
|------|---------|-----------|
| 1. 需求分析 | PRD 评审 | SMART 验收标准、竞品分析 ≥5 |
| 2. 架构设计 | 架构评审 | 接口定义完整、技术选型有依据 |
| 3. UI 设计 | 设计评审 | 风格指南完整、符合技术约束 |
| 4. 测试设计 | 测试评审 | 5类场景覆盖、自动化方案可行 |
| 5. 任务分解 | 计划评审 | 任务粒度 <2天、依赖无循环 |
| 6. 开发实现 | 代码审查 | 单元测试 >80%、零容忍无违规 |
| 7. 测试验证 | 测试报告 | P0/P1 缺陷修复、性能达标 |
| 8. 发布评审 | 发布决策 | 多角色签字、回滚方案就绪 |

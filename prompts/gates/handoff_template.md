# 交接协议模板 (Handoff Protocol Template)

## 概述

交接协议定义了阶段间角色交接的标准化流程和文档格式。确保信息不丢失、责任可追溯、风险有预警。

## 交接原则

1. **文档化交接**：所有交接必须有书面记录
2. **双向确认**：交接方和接收方都必须签字
3. **风险透明**：已知风险和注意事项必须明确传达
4. **信心量化**：交接方必须给出信心度评分

## 交接协议格式

```markdown
# 交接协议

## 基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | {项目名} |
| 交接阶段 | 阶段 {N} → 阶段 {N+1} |
| 交接方 | {角色名} |
| 接收方 | {角色名} |
| 交接日期 | {日期} |
| 交接原因 | 阶段完成，进入下一阶段 |

## 交付物清单

| 序号 | 交付物名称 | 格式 | 位置 | 状态 | 备注 |
|------|-----------|------|------|------|------|
| 1 | {交付物1} | {格式} | {路径} | ✅ 完成 | |
| 2 | {交付物2} | {格式} | {路径} | ✅ 完成 | |
| 3 | {交付物3} | {格式} | {路径} | ⚠️ 部分完成 | {说明} |

## 关键决策记录

| 决策 | 选项 | 最终选择 | 理由 | 影响范围 |
|------|------|---------|------|---------|
| {决策1} | A/B/C | A | {理由} | {范围} |
| {决策2} | A/B | B | {理由} | {范围} |

## 注意事项

1. {注意事项1}
2. {注意事项2}
3. {注意事项3}

## 风险提示

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| {风险1} | 高/中/低 | 高/中/低 | {措施} |
| {风险2} | 高/中/低 | 高/中/低 | {措施} |

## 未完成项

| 序号 | 未完成项 | 原因 | 计划完成时间 | 影响评估 |
|------|---------|------|-------------|---------|
| 1 | {未完成项1} | {原因} | {时间} | {影响} |

## 信心度评估

| 评估维度 | 评分 (1-5) | 说明 |
|---------|-----------|------|
| 交付物完整性 | {1-5} | {说明} |
| 质量满意度 | {1-5} | {说明} |
| 风险可控性 | {1-5} | {说明} |
| **综合信心度** | **{1-5}** | **{说明}** |

### 信心度评分标准
- **5**：非常确定，已充分验证，无已知风险
- **4**：较确定，有理论依据，风险可控
- **3**：一般，需要进一步验证，有中等风险
- **2**：不确定，有较大风险，需要关注
- **1**：很不确定，有严重风险，建议重新评估

## 签字确认

### 交接方确认
- 姓名/角色：{交接方}
- 确认声明：我确认以上信息真实完整，已如实披露所有已知风险
- 签字日期：{日期}

### 接收方确认
- 姓名/角色：{接收方}
- 确认声明：我确认已接收以上交付物，理解注意事项和风险
- 签字日期：{日期}
```

## 8 阶段交接关系

```
阶段1 → 阶段2：产品经理 → 架构师
  交接物：PRD、用户故事、验收标准、竞品分析

阶段2 → 阶段3：架构师 → UI 设计师
  交接物：架构约束、技术栈限制、接口契约

阶段2 → 阶段4：架构师 → 测试专家
  交接物：架构设计、接口定义、非功能性需求

阶段3 → 阶段4：UI 设计师 → 测试专家
  交接物：UI 设计稿、交互流程

阶段2+3+4 → 阶段5：架构师+UI设计师+测试专家 → 独立开发者
  交接物：架构设计、UI设计、测试策略、质量门禁

阶段5 → 阶段6：独立开发者 → 独立开发者（自己）
  交接物：任务分解清单、开发计划

阶段6 → 阶段7：独立开发者 → 测试专家
  交接物：功能代码、单元测试、代码文档

阶段7 → 阶段8：测试专家 → 架构师+产品经理
  交接物：测试报告、质量评估、发布建议
```

## 交接异常处理

### 交接被拒绝
1. 接收方必须书面说明拒绝原因
2. 交接方根据原因修复
3. 重新执行交接流程
4. 如果3次交接失败，升级到多角色共识

### 部分交接
1. 明确哪些交付物已完成、哪些未完成
2. 未完成项必须有计划完成时间和影响评估
3. 接收方确认是否可以在部分交付物缺失的情况下开始工作
4. 未完成项必须跟踪到完成

### 紧急交接
1. 适用场景：紧急 Bug 修复、关键路径阻塞
2. 允许简化交接文档，但必须包含：交付物清单、风险提示、信心度
3. 事后必须补充完整交接文档

# 阶段6：开发实现 (Development Implementation)

## 阶段概述

开发实现是项目生命周期的第六阶段，由独立开发者主导。本阶段的核心目标是将设计文档和任务清单转化为高质量、可运行的代码。

## 主导角色

**独立开发者** (Solo Coder)

## 参与角色

- 架构师：解答架构疑问，审批架构变更
- 测试专家：协助编写单元测试
- UI 设计师：解答设计实现问题

## 入口条件

- [ ] 任务分解完成
- [ ] 开发环境就绪
- [ ] 依赖服务可用（或 mock 方案已确认）

## 上游依赖

- 阶段2 交付物：架构设计文档、接口定义
- 阶段3 交付物：UI 设计稿、组件规格（如有）
- 阶段4 交付物：测试策略、质量门禁
- 阶段5 交付物：任务分解清单、开发计划

## 本阶段交付物

| 交付物 | 负责角色 | 格式要求 | 完成标准 |
|--------|---------|---------|---------|
| 功能代码 | 独立开发者 | 项目代码 | 完整实现，非半成品 |
| 单元测试 | 独立开发者 | 测试代码 | 覆盖率 > 80% |
| 集成测试 | 独立开发者 | 测试代码 | 关键路径覆盖 |
| 代码文档 | 独立开发者 | Markdown | API 文档 + 关键逻辑 |
| 变更日志 | 独立开发者 | Markdown | 按标准格式记录 |

## 阶段执行指引

### 步骤1：环境准备
- 配置开发环境
- 拉取最新代码
- 确认依赖服务可用

### 步骤2：编码实现
- 按任务清单顺序开发
- 遵循零容忍清单（10项绝对禁止）
- 遵循编码规范
- 每完成一个任务，编写对应单元测试

### 步骤3：自测验证
- 运行所有单元测试
- 运行集成测试
- 检查完整性（4维度）
- 修复发现的问题

### 步骤4：代码审查
- 提交代码审查请求
- 回复审查意见
- 修改代码直到审查通过

### 步骤5：文档更新
- 更新 API 文档
- 更新变更日志
- 确保文档与代码一致

## 出口条件（门禁）

- [ ] 所有功能代码完整实现（非 mock/半成品）
- [ ] 单元测试覆盖率 > 80%
- [ ] 所有单元测试通过
- [ ] 代码审查通过
- [ ] 无 P0/P1 级代码质量问题
- [ ] 文档与代码一致

## 下游交接

交接给：**阶段7 - 测试验证（测试专家）**
交接物：功能代码、单元测试、集成测试、代码文档、变更日志

## 常见风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 需求理解偏差 | 返工 | 与产品经理确认 |
| 技术实现困难 | 延期 | 早期 POC 验证 |
| 代码质量低 | 缺陷多 | 零容忍清单 + 代码审查 |
| 测试覆盖不足 | 缺陷遗漏 | 强制 > 80% 覆盖率 |

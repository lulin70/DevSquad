# 阶段2：架构设计 (Architecture Design)

## 阶段概述

架构设计是项目生命周期的第二阶段，由架构师主导。本阶段的核心目标是明确"怎么做"，将产品需求转化为可执行的技术方案。

## 主导角色

**架构师** (Architect)

## 参与角色

- 产品经理：确认架构覆盖所有需求
- 独立开发者：评估技术可行性
- 测试专家：评估可测试性

## 入口条件

- [ ] PRD 文档已评审通过
- [ ] 用户故事和验收标准已确认
- [ ] 产品经理已签字交付

## 上游依赖

- 阶段1 交付物：PRD 文档、用户故事地图、验收标准、竞品分析

## 本阶段交付物

| 交付物 | 负责角色 | 格式要求 | 完成标准 |
|--------|---------|---------|---------|
| 系统架构图 | 架构师 | Mermaid | 覆盖所有模块和交互 |
| 模块职责清单 | 架构师 | Markdown 表格 | 每个模块有输入/输出/异常 |
| 接口定义文档 | 架构师 | Markdown/OpenAPI | 完整的 API 契约 |
| 数据模型设计 | 架构师 | Mermaid ER + 表格 | 含数据字典 |
| 部署架构说明 | 架构师 | Mermaid + Markdown | 含环境规划 |
| 技术选型报告 | 架构师 | Markdown | 含对比分析和决策理由 |
| 非功能性需求方案 | 架构师 | Markdown | 性能/安全/可用性/扩展性 |
| **项目目录结构规范** | 架构师 | Markdown 目录树 | 含命名规范、模块边界、.gitignore |

## 阶段执行指引

### 步骤1：需求理解
- 深入阅读 PRD，理解业务需求
- 与产品经理确认需求细节
- 识别技术难点和风险

### 步骤2：架构设计
- 使用系统性思维回答4个关键问题
- 设计系统架构（分层/微服务/单体）
- 定义模块边界和职责
- 设计数据模型

### 步骤3：技术选型
- 列出候选技术方案
- 从功能、性能、生态、团队熟悉度4个维度对比
- 输出选型决策和理由

### 步骤4：接口定义
- 定义模块间接口契约
- 包含输入/输出/异常/版本
- 前后端接口对齐

### 步骤5：定义项目目录结构
- 根据项目类型选择基础结构模板（Web前端/后端/全栈/Python/AI-ML）
- 根据模块设计定制目录结构
- 定义命名规范（文件/目录/组件）
- 定义模块边界和依赖方向约束
- 定义 .gitignore 规则

### 步骤6：架构评审
- 组织产品经理和独立开发者参与评审
- 确认架构覆盖所有需求
- 确认技术方案可行

## 出口条件（门禁）

- [ ] 架构设计覆盖 PRD 所有功能需求
- [ ] 模块间接口定义完整（输入/输出/异常/版本）
- [ ] 非功能性需求有明确技术方案
- [ ] 技术选型有对比分析和决策理由
- [ ] 部署架构已说明
- [ ] **项目目录结构已定义（含命名规范和模块边界约束）**
- [ ] 架构评审通过（至少产品经理 + 独立开发者签字）

## 下游交接

交接给：**阶段3 - UI 设计（UI 设计师）** 和 **阶段4 - 测试设计（测试专家）**
交接物：架构设计文档、接口定义、技术约束说明

## 常见风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 过度设计 | 开发效率低 | 遵循 YAGNI 原则 |
| 设计不足 | 后期重构 | 5-Why 分析法 |
| 技术选型失误 | 项目延期 | 对比分析 + POC 验证 |
| 接口定义不完整 | 集成困难 | 严格接口契约 |

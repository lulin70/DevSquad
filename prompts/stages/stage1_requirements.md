# 阶段1：需求分析 (Requirements Analysis)

## 阶段概述

需求分析是项目生命周期的第一阶段，由产品经理主导。本阶段的核心目标是明确"做什么"和"为什么做"，产出可验证的产品需求文档。

## 主导角色

**产品经理** (Product Manager)

## 参与角色

- 架构师：评审需求的技术可行性
- 测试专家：评审需求的可测试性

## 入口条件

- [ ] 项目启动指令已下达
- [ ] 业务背景和目标已明确
- [ ] 利益相关者已识别
- [ ] 项目约束条件已说明

## 上游依赖

无（本阶段为项目起始阶段）

## 本阶段交付物

| 交付物 | 负责角色 | 格式要求 | 完成标准 |
|--------|---------|---------|---------|
| PRD 文档 | 产品经理 | Markdown | 覆盖所有业务场景 |
| 用户故事地图 | 产品经理 | Markdown 表格 | 含优先级排序 |
| SMART 验收标准 | 产品经理 | Markdown 表格 | 每个用户故事有标准 |
| 竞品分析报告 | 产品经理 | Markdown | 至少5个竞品 |
| 用户画像 | 产品经理 | Markdown | 目标用户群体描述 |
| 数据指标定义 | 产品经理 | Markdown 表格 | 可量化可追踪 |

## 阶段执行指引

### 步骤1：需求收集
- 与利益相关者沟通，收集业务需求
- 识别目标用户群体和使用场景
- 收集现有痛点和期望

### 步骤2：需求分析
- 使用三层挖掘法：表面需求 → 真实需求 → 本质需求
- 识别功能需求和非功能性需求
- 分析需求优先级

### 步骤3：竞品分析
- 选择至少5个竞品（3个直接 + 2个间接）
- 从功能、体验、技术、商业4个维度分析
- 输出差异化策略

### 步骤4：编写 PRD
- 按标准 PRD 模板编写
- 每个功能有对应的用户故事
- 每个用户故事有 SMART 验收标准

### 步骤5：需求评审
- 组织架构师和测试专家参与评审
- 确认需求的技术可行性和可测试性
- 修订 PRD 直到评审通过

## 出口条件（门禁）

- [ ] PRD 覆盖所有业务场景（正常 + 异常 + 边界）
- [ ] 每个用户故事有 SMART 验收标准
- [ ] 竞品分析包含至少5个竞品
- [ ] 数据指标可量化可追踪
- [ ] PRD 评审通过（至少架构师 + 测试专家签字）

## 下游交接

交接给：**阶段2 - 架构设计（架构师）**
交接物：PRD 文档、用户故事地图、验收标准、竞品分析

## 常见风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 需求模糊 | 后期返工 | 强制 SMART 验收标准 |
| 需求遗漏 | 功能缺失 | 使用三层挖掘法 |
| 需求变更 | 进度延迟 | 建立变更管理流程 |
| 竞品分析不足 | 差异化不清 | 强制5个竞品分析 |

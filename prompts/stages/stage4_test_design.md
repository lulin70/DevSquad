# 阶段4：测试设计 (Test Design)

## 阶段概述

测试设计是项目生命周期的第四阶段，由测试专家主导。本阶段的核心目标是建立全面的质量保障体系，确保产品在每个维度都可验证。

## 主导角色

**测试专家** (Test Expert)

## 参与角色

- 产品经理：确认测试覆盖所有验收标准
- 架构师：确认测试方案的技术可行性
- 独立开发者：确认自动化测试框架可行

## 入口条件

- [ ] 架构设计评审通过
- [ ] UI 设计评审通过（如有）
- [ ] PRD 验收标准已确认

## 上游依赖

- 阶段1 交付物：PRD 文档、验收标准
- 阶段2 交付物：架构设计文档、接口定义
- 阶段3 交付物：UI 设计稿、交互流程（如有）

## 本阶段交付物

| 交付物 | 负责角色 | 格式要求 | 完成标准 |
|--------|---------|---------|---------|
| 测试策略文档 | 测试专家 | Markdown | 测试金字塔 + 类型分配 |
| 测试用例 | 测试专家 | Markdown 表格 | 5类场景全覆盖 |
| 自动化测试方案 | 测试专家 | Markdown | 框架选型 + 覆盖率目标 |
| 质量门禁定义 | 测试专家 | Markdown | 各阶段质量标准 |
| 测试环境需求 | 测试专家 | Markdown | 环境规格和数据 |

## 阶段执行指引

### 步骤1：测试策略制定
- 确定测试金字塔比例（70/20/10）
- 确定测试类型分配（单元/集成/E2E/性能/安全）
- 确定测试优先级

### 步骤2：测试用例设计
- 正常场景：覆盖所有 Happy Path
- 异常场景：错误输入、网络故障、服务不可用
- 边界场景：极值、空值、最大最小值
- 性能场景：并发、大数据量、长时间运行
- 安全场景：注入、越权、数据泄露

### 步骤3：自动化方案
- 选择测试框架
- 设计测试数据管理方案
- 定义覆盖率目标（单元 >80%，集成 >60%）

### 步骤4：质量门禁定义
- 代码提交门禁：单元测试通过
- 代码合并门禁：代码审查通过
- 阶段完成门禁：测试覆盖率达标
- 发布门禁：所有 P0/P1 缺陷修复

### 步骤5：测试计划评审
- 组织架构师和独立开发者参与评审
- 确认测试方案可行
- 确认质量门禁合理

## 出口条件（门禁）

- [ ] 测试用例覆盖所有验收标准
- [ ] 5类场景（正常/异常/边界/性能/安全）全覆盖
- [ ] 自动化测试方案可行（框架已验证）
- [ ] 质量门禁定义明确（可量化）
- [ ] 测试计划评审通过（至少架构师 + 独立开发者签字）

## 下游交接

交接给：**阶段5 - 任务分解（独立开发者）**
交接物：测试策略、测试用例、质量门禁、自动化测试框架

## 常见风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 测试覆盖不足 | 缺陷遗漏 | 5类场景强制覆盖 |
| 自动化成本高 | 进度延迟 | 优先自动化核心路径 |
| 测试环境不一致 | 结果不可靠 | 容器化测试环境 |
| 质量门禁过严 | 流程阻塞 | 门禁标准可协商 |

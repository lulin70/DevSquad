# 阶段5：任务分解 (Task Breakdown)

## 阶段概述

任务分解是项目生命周期的第五阶段，由独立开发者主导。本阶段的核心目标是将架构设计和测试要求转化为可执行的开发任务清单。

## 主导角色

**独立开发者** (Solo Coder)

## 参与角色

- 架构师：确认任务分解符合架构设计
- 测试专家：确认任务包含测试要求

## 入口条件

- [ ] 架构设计评审通过
- [ ] 测试设计评审通过
- [ ] UI 设计评审通过（如有）

## 上游依赖

- 阶段2 交付物：架构设计文档、接口定义
- 阶段3 交付物：UI 设计稿、组件规格（如有）
- 阶段4 交付物：测试策略、质量门禁

## 本阶段交付物

| 交付物 | 负责角色 | 格式要求 | 完成标准 |
|--------|---------|---------|---------|
| 任务分解清单 | 独立开发者 | Markdown 表格 | 每个任务有输入/输出/验收标准 |
| 任务依赖关系图 | 独立开发者 | Mermaid | 依赖清晰无循环 |
| 开发计划 | 独立开发者 | Markdown | 含里程碑和时间线 |
| 技术风险清单 | 独立开发者 | Markdown | 每个风险有缓解措施 |

## 阶段执行指引

### 步骤1：理解设计
- 深入阅读架构设计文档
- 理解接口定义和数据模型
- 确认技术栈和框架版本

### 步骤2：任务识别
- 按模块划分开发任务
- 每个任务对应明确的交付物
- 识别任务间依赖关系

### 步骤3：任务细化
- 单个任务工作量 < 2天
- 每个任务有明确的输入/输出
- 每个任务有验收标准

### 步骤4：计划排期
- 按依赖关系排序
- 识别关键路径
- 设置里程碑

### 步骤5：计划评审
- 组织架构师参与评审
- 确认任务分解合理
- 确认工作量估算合理

## 出口条件（门禁）

- [ ] 每个任务有明确的输入/输出/验收标准
- [ ] 任务粒度合理（单个任务 < 2天）
- [ ] 依赖关系清晰无循环
- [ ] 工作量估算合理
- [ ] 开发计划评审通过

## 下游交接

交接给：**阶段6 - 开发实现（独立开发者自己）**
交接物：任务分解清单、依赖关系图、开发计划

## 常见风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 任务粒度过大 | 进度不可控 | 强制 < 2天 |
| 依赖遗漏 | 阻塞开发 | 依赖关系图检查 |
| 工作量低估 | 延期 | 预留缓冲时间 |
| 技术风险未识别 | 返工 | 技术风险清单 |
